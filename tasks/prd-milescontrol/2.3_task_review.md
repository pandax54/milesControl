### Minor Issues

**1. DB query fetches all subscriptions instead of only active ones (accrual-projector.service.ts, line 200)**

The `fetchUserProjection` function fetches all subscriptions for a user (`where: { userId }`) without filtering by status. While `calculateMilesForMonth` correctly returns 0 for non-ACTIVE subscriptions, this causes unnecessary data transfer from the database. For users with many historical/cancelled subscriptions, this is wasted work.

Suggested fix:
```typescript
prisma.clubSubscription.findMany({
  where: { userId, status: 'ACTIVE' },
  // ...
}),
```

Note: If the intent is to show cancelled/paused subscriptions in the projection breakdown (e.g., greyed out), keep the current approach but add a comment explaining why.

**2. Log message at line 231 says "Fetching" but projection is already computed**

The `logger.info` call at line 231 says "Fetching user accrual projection" but it runs after the data has been fetched and the projection is about to be computed. The message is slightly misleading.

Suggested fix:
```typescript
logger.info(
  { userId, subscriptionCount: subscriptionsForProjection.length, totalBalance, monthsAhead },
  'Computing user accrual projection',
);
```

**3. `parseAccrualSchedule` silently drops invalid phases without logging (accrual-projector.service.ts, line 181-186)**

When individual phases fail Zod validation, they are silently skipped. This is a reasonable defensive approach, but a `warn` log would help diagnose data quality issues in production -- e.g., if a subscription's accrual schedule was corrupted.

Suggested fix:
```typescript
if (!parsed.success) {
  logger.warn({ item, errors: parsed.error.flatten() }, 'Skipping invalid accrual phase');
  continue;
}
```

## Recommendations

1. **Filter active subscriptions at the DB level** -- Add `status: 'ACTIVE'` to the Prisma query in `fetchUserProjection` to avoid fetching cancelled/expired subscriptions unnecessarily. This is a minor optimization but becomes meaningful as users accumulate historical subscriptions.

2. **Add a warning log in `parseAccrualSchedule`** when phases fail validation -- this will help diagnose data quality issues without breaking the projection flow.

3. **Fix the log message timing** -- "Fetching" should be "Computing" since the data has already been fetched at that point, or move the log to before the `Promise.all` call.

The implementation is well-crafted, thoroughly tested, and meets all task requirements. The pure function design enables easy testing and future reuse. The three minor observations are non-blocking improvements that can be addressed in a follow-up. The code is production-ready.
