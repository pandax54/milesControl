## Key Implementation Details Verified

### ✅ Watchlist Service (`flight-watchlist.service.ts`)
- **CRUD operations**: `listWatchlistItems()`, `listActiveWatchlistItems()`, `getWatchlistItem()`, `createWatchlistItem()`, `updateWatchlistItem()`, `toggleWatchlistItem()`, `deleteWatchlistItem()` — all with proper authorization checks.
- **Price checking**: `checkWatchlistItemPrices()` integrates with `searchFlights()`, creates notifications, and handles both miles and cash targets.
- **Helper function**: `resolveDepartureDate()` correctly prioritizes `earliestDate` if >= today, falls back to `latestDate`, returns null if both are in the past.

### ✅ Validation (`watchlist.schema.ts`)
- `createWatchlistItemSchema`: IATA codes (3 uppercase letters), date strings (YYYY-MM-DD), 1–9 passengers, at least one price target.
- `updateWatchlistItemSchema`: All fields optional except watchlistId, supports nullable targets for clearing.
- Proper `.refine()` check ensuring `targetMilesPrice || targetCashPrice` is not null.

### ✅ Server Actions (`watchlist.ts`)
- Input validation via Zod `.safeParse()`.
- Authentication via `requireUserId()` with proper error handling.
- Revalidation of `/flights/watchlist` after mutations.
- Distinction between validation errors (`Invalid input`) and domain errors (`Watchlist item not found`).

### ✅ UI Components
- **AddWatchlistForm**: Client-side validation with specific error messages, form reset on success, loading state during transition.
- **WatchlistItemCard**: Displays route, cabin, dates, targets, last checked time. Pause/Resume toggle and delete button with loading states.
- **WatchlistPage**: Conditional rendering (form visible when empty or toggled, card grid when populated), empty state with icon and description.
- **Route page**: RSC pattern, auth redirect to login, server-side data fetch via `listWatchlistItems()`.

### ✅ Test Coverage
- **Service tests** (491 lines): All CRUD functions tested for success and error cases. Price checking tested with miles/cash targets, past dates, and alert creation.
- **Action tests** (251 lines): All server actions tested for validation, authentication, error handling, and revalidation.
- **Mocking**: Only external boundaries (Prisma, logger, flight search) are mocked. Service logic is not mocked in action tests, ensuring integration coverage.
