## Summary

Task 3.3 implements a concrete scraper for the Pontos Pra Voar blog, extending `BaseScraper`. The scraper targets `/category/promocoes/`, parses `article` elements, and extracts promotion metadata from titles, categories, dates, and excerpts. It handles both relative Portuguese dates ("2 horas atrás") and absolute Portuguese dates ("19 de março de 2026") with accented character support. The implementation reuses shared `promotion-helpers` for type classification and program extraction. Code quality is excellent, consistent with the patterns established in tasks 3.1 and 3.2. 43 tests cover all extraction helpers and integration scenarios.

## ✅ Positive Highlights

1. **Robust dual date parser** — `parsePtDate` cleanly handles two distinct formats: relative Portuguese dates ("X horas/minutos/dias atrás") and absolute Portuguese dates ("19 de março de 2026"). Uses a well-structured `RELATIVE_TIME_PATTERNS` array with typed `unit` discriminants, avoiding a chain of if/else blocks. The `PT_MONTHS` lookup table maps all 12 Portuguese month names with proper accented character support via `[\wÀ-ú]+` in the regex.

2. **Consistent scraper architecture** — Follows the exact same structural pattern as PdP and MC: constructor with `Partial<ScraperConfig>` overrides, `extractPromotions` override returning `ScrapedPromotion[]`, site-specific helpers exported individually, and structured Pino logging with `{ articlesFound }`.

3. **Full reuse of shared promotion helpers** — Correctly imports and uses `classifyPromoType`, `extractBonusPercent`, `extractDiscountPercent`, and `extractPrograms` from `./promotion-helpers`, validating the refactoring done in task 3.2.

4. **Pure, individually testable extraction functions** — `extractCategory`, `extractDateText`, `extractRawContent`, and `parsePtDate` are all pure exported functions with their own dedicated test suites (4, 5, 3, and 13 tests respectively).

5. **Excellent test coverage** — 43 tests pass. Statement, function, and line coverage are at 100%. Branch coverage is at 94.11%, with only two defensive fallback branches uncovered (line 78: `attr('href') ?? ''`; line 172: `Number.isNaN(date.getTime())` for invalid absolute dates). These are reasonable safety guards.

6. **Well-crafted test fixtures** — `buildArticleHtml` accurately reproduces the Pontos Pra Voar DOM structure (article > entry-meta > fancy-categories + date links, h3.entry-title > a, entry-excerpt), making integration tests realistic and readable.

7. **Proper use of `vi.useFakeTimers()` for time-dependent tests** — The relative date parsing test correctly uses `vi.setSystemTime` and calls `vi.useRealTimers()` in cleanup, following the testing standards.

8. **Clean `extractDateText` implementation** — Iterates `.entry-meta a` links, skips `fancy-categories` links, and validates text content against known date patterns before accepting it. Avoids false positives from author links or other meta elements.

9. **Typed mock factory** — `buildMockScraperRun` uses proper `ScraperRun` type from Prisma, consistent with the project-wide convention. No `as never` casts anywhere in the test file.

10. **Proper `readonly` typing** — `RELATIVE_TIME_PATTERNS` uses `readonly` on both the array and each element's properties, and the `unit` property uses a union type (`'minutes' | 'hours' | 'days'`) for type safety in `subtractTime`.

11. **No `any` types, no `console.log`, no `process.env` access** — Clean TypeScript throughout. All constants use `SCREAMING_SNAKE_CASE`. Pino logging with structured data.
