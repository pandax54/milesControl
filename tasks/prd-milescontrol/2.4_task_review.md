## Summary

Task 2.4 implements a full CRUD for credit cards: add, edit, and remove cards with bank name, card name, points program, earning rates, annual fee, waived-fee flag, and benefits. The implementation spans four layers:

1. **Zod validators** (`credit-card.schema.ts`) for create, update, and delete inputs.
2. **Service layer** (`credit-card.service.ts`) with `listCreditCards`, `createCreditCard`, `updateCreditCard`, `deleteCreditCard`, and a custom `CreditCardNotFoundError`.
3. **Server actions** (`credit-cards.ts`) with `addCreditCard`, `editCreditCard`, `removeCreditCard` -- each validates input, checks authentication, delegates to the service, and revalidates the page cache.
4. **UI layer**: a server-rendered page grouping cards by bank, a create dialog, an edit dialog, and a delete confirmation dialog.

Overall quality is high. The code is well-structured, follows project standards, uses Pino for logging (no console.log), validates all inputs with Zod, and has thorough test coverage. All 212 tests pass, TypeScript compiles cleanly, and coverage exceeds the 80% threshold. A few minor observations are noted below.

### Minor Issues

1. **Duplicated form logic between create and edit dialogs**
   - Files: `src/components/dashboard/credit-card-form-dialog.tsx` and `src/components/dashboard/edit-credit-card-dialog.tsx`
   - The two dialog components duplicate approximately 80% of their form markup and validation logic (benefit add/remove, numeric parsing, field layout). This is not a blocking problem, but extracting a shared `CreditCardFormFields` component would reduce maintenance burden and eliminate the risk of the two forms diverging over time.

2. **Many individual `useState` calls instead of a form state object or form library**
   - File: `src/components/dashboard/credit-card-form-dialog.tsx`, lines 21-29
   - Nine separate `useState` calls are used for form state. While functional, a single state object or a lightweight form hook (e.g., `useReducer` or `react-hook-form` with Zod resolver) would reduce boilerplate and make reset logic cleaner. This is a style preference, not a rule violation.

3. **`as never` type casts in test mocks**
   - File: `src/lib/services/credit-card.service.test.ts`, lines 62, 87, 110, 146, 148, 161, 177, 189, 230, 231
   - The mocks use `as never` to satisfy Prisma's complex return types. While this works and is a common workaround for Prisma in tests, a cleaner approach would be to define typed factory helpers that return objects conforming to the actual Prisma model type. Non-blocking.

4. **`annualFee` type mismatch between Prisma `Decimal` and component `number`**
   - File: `src/components/dashboard/credit-card-card.tsx`, line 16
   - The `CreditCardData` interface declares `annualFee: Decimal | number`, and the component has runtime coercion at line 34 (`typeof card.annualFee === 'number' ? ...`). The page (line 59) passes the raw Prisma object which uses `Decimal`. This works but is fragile. Consider normalizing `annualFee` to `number` at the page boundary (with `Number(card.annualFee)`) so the component interface is simpler and does not depend on Prisma's `Decimal` type leaking into the UI layer.

5. **Edit dialog does not reset form state when card prop changes**
   - File: `src/components/dashboard/edit-credit-card-dialog.tsx`
   - The `useState` initializers (lines 28-38) capture the card prop values only on initial mount. If React re-renders the parent with updated card data (e.g., after another user's edit in a future real-time scenario, or after the component stays mounted across navigation), the form fields will show stale values. This is unlikely to be a problem today given the server-component architecture and `revalidatePath`, but worth noting for future robustness.

6. **Empty state renders `CreditCardFormDialog` twice**
   - File: `src/app/(dashboard)/credit-cards/page.tsx`, lines 36 and 46
   - When there are no cards, both the top-level "Add Card" button and the one inside the empty-state block are rendered. This is intentional UX (the empty state CTA is more prominent), but consider whether the top-level button should be conditionally hidden when showing the empty state to avoid visual redundancy.

## Recommendations

1. **Extract shared form fields**: Create a `CreditCardFormFields` component to deduplicate the create and edit dialog forms. This is the highest-value improvement.
2. **Normalize Decimal at the boundary**: Convert `annualFee` from `Decimal` to `number` in the page server component before passing to client components, removing the Prisma type dependency from the UI.
3. **Consider form library**: If forms grow more complex (e.g., adding validation messages per field), adopting `react-hook-form` with `@hookform/resolvers/zod` would reduce boilerplate significantly.
4. **Type-safe test factories**: Replace `as never` casts with typed factory functions that return proper Prisma model shapes, improving test maintainability.
