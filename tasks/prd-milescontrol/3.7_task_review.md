### 🟡 Major Issues

1. **Duplicate HTML `id` attributes in compare mode** -- `src/components/dashboard/calculator-form.tsx`, lines 64-133.

   The `CalculatorFields` component uses hardcoded `id` attributes (`id="purchasePricePerPoint"`, `id="quantity"`, etc.) and matching `htmlFor` attributes on `Label` elements. In the `CompareCalculator`, multiple instances of `CalculatorFields` render simultaneously (2-3 scenarios), producing duplicate `id` values in the DOM. This violates the HTML specification (IDs must be unique per document), breaks `Label`-to-`Input` associations (clicking a label in Scenario 2 may focus the input in Scenario 1), and causes accessibility screen reader issues.

   **Suggested fix**: Accept a `prefix` prop in `CalculatorFields` and use it to namespace the IDs:

   ```tsx
   interface CalculatorFieldsProps {
     form: FormValues;
     onChange: (field: keyof FormValues, value: string) => void;
     idPrefix?: string;
   }

   function CalculatorFields({ form, onChange, idPrefix = '' }: CalculatorFieldsProps) {
     const fieldId = (name: string) => idPrefix ? `${idPrefix}-${name}` : name;
     return (
       <div className="grid gap-4 sm:grid-cols-2">
         <div className="space-y-2">
           <Label htmlFor={fieldId('purchasePricePerPoint')}>Price per Point (R$)</Label>
           <Input id={fieldId('purchasePricePerPoint')} ... />
         </div>
         {/* ... same pattern for all fields */}
       </div>
     );
   }
   ```

   Then in `CompareCalculator`, pass `idPrefix={`scenario-${scenario.id}`}`.

2. **Module-level mutable counter leaks across component lifecycles** -- `src/components/dashboard/calculator-form.tsx`, line 247.

   `let scenarioIdCounter = 2` is declared at module scope outside any component. It persists across React renders and even across unmount/remount cycles of `CompareCalculator`. For example: user adds a scenario (counter becomes 3), switches to the single tab and back -- the counter is still 3 but `handleReset` resets it to 2. More subtly, in a server-rendered Next.js app, module-level state can persist across different user requests in the same server process if the module is cached (though this is a client component, so the risk is limited to the browser session).

   **Suggested fix**: Use `useRef` to keep the counter scoped to the component instance:

   ```tsx
   function CompareCalculator() {
     const scenarioIdRef = useRef(2);
     // ...
     function addScenario() {
       if (scenarios.length >= MAX_SCENARIOS) return;
       scenarioIdRef.current += 1;
       setScenarios((prev) => [
         ...prev,
         { id: scenarioIdRef.current, name: `Scenario ${prev.length + 1}`, values: EMPTY_FORM },
       ]);
     }
     function handleReset() {
       setScenarios([...]);
       setComparison(null);
       setError(null);
       scenarioIdRef.current = 2;
     }
   }
   ```

### 🟢 Minor Issues

1. **Error silently swallowed in `PromoCalculatorEmbed`** -- `src/components/dashboard/promo-calculator-embed.tsx`, lines 33-47.

   When `calculate` returns `{ success: false }`, the `handleCalculate` function does nothing -- it neither shows an error message nor clears the previous result. The user clicks "Calculate" and nothing visually changes. Compare this with `SingleCalculator` which properly displays `error` state. Consider adding an `error` state and showing it in the UI, or at minimum clearing the previous result on failure.

2. **Server action name `calculate` is generic** -- `src/actions/calculator.ts`, line 31.

   The exported function `calculate` is a very generic name for a server action. Per code standards, function names should start with a verb and clearly describe what they do. Consider renaming to `calculateCostPerMilheiroAction` to match the pattern established by `compareScenariosAction` and to avoid potential naming collisions as the codebase grows. This also makes imports self-documenting: `import { calculateCostPerMilheiroAction } from '@/actions/calculator'`.

3. **Unused `index` parameter in `CompareCalculator` map** -- `src/components/dashboard/calculator-form.tsx`, line 321.

   The `scenarios.map((scenario, index) => ...)` receives `index` but it is not used inside the iteration. The `scenario.id` is used for the key and for event handlers. The `index` parameter can be removed.

4. **`CalculationDisplayData` duplicates `CostCalculation` interface** -- `src/components/dashboard/calculator-result-card.tsx`, lines 8-13.

   The `CalculationDisplayData` interface has the exact same shape as `CostCalculation` from the service. It re-declares `totalCost`, `totalMiles`, `costPerMilheiro`, and `rating` with `readonly`. While creating a UI-specific type can be a valid layer boundary, in this case the fields and types are identical. Consider importing `CostCalculation` directly or using `Pick<CostCalculation, ...>` to avoid drift between the two definitions.

## Recommendations

1. **Fix duplicate HTML IDs (major)**: Add an `idPrefix` prop to `CalculatorFields` and use it to namespace form element IDs in compare mode. This is the highest-priority fix as it affects accessibility and label-input association correctness.

2. **Move `scenarioIdCounter` into a `useRef` (major)**: Replace the module-level mutable variable with a `useRef` inside `CompareCalculator` to scope the counter to the component instance and avoid stale state across unmount/remount cycles.

3. **Add error feedback to `PromoCalculatorEmbed` (minor)**: Show an error message when calculation fails in the embed component, or at minimum clear the previous result so the user knows something went wrong.

4. **Consider renaming `calculate` to `calculateCostPerMilheiroAction` (minor)**: Aligns with the naming pattern of `compareScenariosAction` and makes the import self-documenting.

5. **Future (task 3.8)**: The `PromoCalculatorEmbed` will likely need to integrate with the Miles Value Advisor to show comparisons against the user's historical average cost, per PRD F3.2.
