### 🟢 Minor Issues

1. **Unused `index` parameter in scenario card map** — `src/components/dashboard/calculator-form.tsx`, line 322.

   The `scenarios.map((scenario, index) => ...)` declares `index` but only uses `scenario.id` for keys and event handlers. The `index` variable is unused in this particular map callback. This is non-blocking since TypeScript allows unused callback parameters, but removing it or prefixing with `_` would clarify intent.

## ✅ Positive Highlights

1. **Clean server action architecture**: `calculateCostPerMilheiroAction` and `compareScenariosAction` properly validate with Zod before calling the service layer, use structured error handling with specific error classes, and log errors with Pino using the correct `{ err: error }` data-first pattern.

2. **Comprehensive action tests**: 14 tests cover all paths — valid input, invalid input (negative values, non-integer quantity), service errors, custom error classes (`InsufficientScenariosError`, `TooManyScenariosError`), optional fields, default values, and invalid scenarios within comparison input. Tests follow AAA pattern with builder functions and proper `vi.mock`/`vi.mocked` usage.

3. **Strong separation of concerns**: The page (`page.tsx`) handles auth and layout, the form (`calculator-form.tsx`) handles state and user interaction, the result card (`calculator-result-card.tsx`) handles display, the embed (`promo-calculator-embed.tsx`) provides a compact variant, and the server action (`calculator.ts`) handles validation and service delegation.

4. **Well-designed preset system**: Presets displayed as buttons in single mode and badges in compare mode. The `inputToForm`/`parseFormToInput` conversion functions cleanly handle the string-to-number boundary between form state and calculator input types.

5. **`useTransition` for non-blocking UI**: Both `SingleCalculator` and `CompareCalculator` use React's `useTransition` for server action calls, preventing UI blocking and showing proper pending states.

6. **Compact embed component (PRD F3.5)**: `PromoCalculatorEmbed` uses `useId()` for globally unique IDs, pre-fills fields from promotion data, and now properly shows error feedback when calculation fails.

7. **Named constants throughout**: `MIN_SCENARIOS`, `MAX_SCENARIOS`, `EMPTY_FORM`, `INITIAL_COUNTER`, and `RATING_VARIANTS`/`RATING_LABELS` maps avoid magic values.

8. **Proper auth guard**: The calculator page checks for authenticated session and redirects to `/login`, consistent with all dashboard pages.

9. **Accessibility-correct compare mode**: `idPrefix` on `CalculatorFields` ensures unique HTML IDs across multiple scenario instances, and `aria-label` on remove buttons provides screen reader context.

10. **No `console.log`, no `any`, no `process.env` direct access**: Code is clean of all prohibited patterns.
