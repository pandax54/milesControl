### 🟢 Minor Issues

**1. `BuildPublicPageMetadataOptions.path` type has redundant manual literals**

- **File**: `src/lib/seo/public-pages.ts`, line 12–15
- **Description**: The `path` union type manually repeats the `/_public` values that are already derivable from `PUBLIC_ROUTE_REWRITES`:
  ```ts
  path: keyof typeof PUBLIC_ROUTE_REWRITES | '/_public' | '/_public/calculator' | '/_public/promotions' | '/_public/promotions/calendar';
  ```
  This is fragile — adding a new public route requires updating the map AND the type union.
- **Suggestion**: Derive from the const map:
  ```ts
  path: keyof typeof PUBLIC_ROUTE_REWRITES | (typeof PUBLIC_ROUTE_REWRITES)[keyof typeof PUBLIC_ROUTE_REWRITES];
  ```

**2. `dangerouslySetInnerHTML` in `JsonLdScript` has no clarifying comment**

- **File**: `src/components/public/json-ld-script.tsx`, line 9
- **Description**: `dangerouslySetInnerHTML` is the correct and standard pattern for injecting JSON-LD structured data in Next.js, but it reads as an unsafe pattern to reviewers unfamiliar with the intent. The data is always generated from typed, internal inputs so there is no XSS risk here.
- **Suggestion**: Add a brief comment explaining the intent:
  ```tsx
  {/* JSON-LD structured data — data is generated from typed internal inputs, not user content */}
  <script
    type="application/ld+json"
    dangerouslySetInnerHTML={{ __html: JSON.stringify(data) }}
  />
  ```

## ✅ Positive Highlights

- **Elegant middleware rewrite pattern**: Public pages live under `/_public` for clean isolation but are transparently served at canonical URLs (`/`, `/calculator`, etc.) via `getPublicRewritePath` in the proxy. SEO crawlers see clean paths; the internal routing is invisible.
- **Canonical URL correctness**: `buildPublicPageMetadata` correctly strips the `/_public` prefix when generating `alternates.canonical` and `openGraph.url`, ensuring Google sees the public-facing URL even though the page is rendered from `/_public/…`.
- **Centralized SEO config**: `site-config.ts` and `public-pages.ts` keep all constants, URL builders, and schema builders in one place — easy to maintain and extend.
- **Comprehensive structured data**: JSON-LD schemas for `WebPage`, `SoftwareApplication`, and `BreadcrumbList` are implemented with correct `@context` / `@type` conventions and absolute URLs.
- **`robots.ts` is precise**: It explicitly allows only the four public routes and blocks all internal app paths, including `/_public` (the internal rewrite target). This is correct — crawlers should only access the canonical paths.
- **Well-tested SEO helpers**: `public-pages.test.ts`, `robots.test.ts`, and `sitemap.test.ts` cover the key behaviors (rewrite logic, canonical URL generation, sitemap entries, structured data shapes) with clear AAA-pattern tests.
- **Auth config update is minimal and safe**: `PUBLIC_EXACT_ROUTES` and `PUBLIC_PREFIX_ROUTES` cleanly separate exact-match vs prefix-match public paths without altering existing behavior for other routes.
- **No `console.log`, no direct `process.env`**: All logging and config access follows project standards.
- **Timer cleanup in sitemap test**: `vi.useFakeTimers()` / `vi.useRealTimers()` are correctly paired.

## Recommendations

1. **(Minor)** Derive the `path` type in `BuildPublicPageMetadataOptions` from `PUBLIC_ROUTE_REWRITES` to avoid the fragile manual union. This eliminates a maintenance hazard when new public routes are added.

2. **(Minor)** Add a one-line comment to `JsonLdScript` explaining that `dangerouslySetInnerHTML` is intentional and safe (data is generated from typed inputs, not user content). This reduces future reviewer friction.

3. **(Future / Non-blocking)** The `buildSoftwareApplicationSchema` does not include an `offers` or `aggregateRating` field, which can improve rich snippet eligibility. Not required for MVP, but worth considering for SEO uplift.

4. **(Future / Non-blocking)** The `PublicShell` navigation shows three hardcoded links. If new public pages are added, the `PUBLIC_NAV_ITEMS` const already centralizes this — just ensure new pages are added there rather than inline.
