### Minor Issues

1. **Redundant `isUrgent` condition** - `src/components/promotions/deadline-countdown.tsx`, line 34

   The condition `days === 0 && hours < 24` is redundant because when `days === 0`, `hours` is always the remainder of `diff % MILLISECONDS_PER_DAY / MILLISECONDS_PER_HOUR`, which by definition is always less than 24. The check `hours < 24` is tautological.

   Suggested fix: simplify to `const isUrgent = days === 0;` for clarity. This does not change behavior but makes the intent clearer.

2. **Duplicated pure functions in test file** - `src/components/promotions/deadline-countdown.test.ts`, lines 19-48

   The `computeTimeRemaining` and `formatTimeRemaining` functions are copy-pasted into the test file rather than exported from the component and imported. While this works, it means the test is not actually testing the real implementation - it is testing a copy. If the source functions diverge from the test copy, the tests would pass but the component would be broken.

   Suggested fix: export `computeTimeRemaining` and `formatTimeRemaining` from `deadline-countdown.tsx` and import them in the test file:

   ```ts
   // deadline-countdown.tsx
   export function computeTimeRemaining(deadline: Date): TimeRemaining { ... }
   export function formatTimeRemaining(time: TimeRemaining): string { ... }

   // deadline-countdown.test.ts
   import { computeTimeRemaining, formatTimeRemaining } from './deadline-countdown';
   ```

3. **`useEffect` dependency uses `.getTime()` expression** - `src/components/promotions/deadline-countdown.tsx`, line 67

   The dependency `[deadlineDate.getTime()]` is a computed expression inside the dependency array. While functionally correct (avoids unnecessary re-runs when Date identity changes but value stays same), ESLint's `react-hooks/exhaustive-deps` rule would flag this. A `useMemo` for the timestamp value would be more idiomatic:

   ```ts
   const deadlineMs = useMemo(() => deadlineDate.getTime(), [deadlineDate.getTime()]);
   useEffect(() => { ... }, [deadlineMs]);
   ```

   This is cosmetic and does not affect correctness.

4. **Type assertion in filter `handleChange`** - `src/components/promotions/promotion-filters.tsx`, line 35

   The line `(updated as Record<string, string>)[key] = value;` uses a type assertion to work around TypeScript's strict property access. This is a pragmatic solution but bypasses type safety. A slightly more type-safe alternative would use a discriminated approach, though the current approach is contained and the risk is low.

## Recommendations

1. **Export pure functions from countdown component** - Extract and export `computeTimeRemaining` and `formatTimeRemaining` so the test file imports the real implementation rather than maintaining a duplicate copy. This is the most actionable improvement.

2. **Simplify `isUrgent` condition** - Change `days === 0 && hours < 24` to just `days === 0` for clarity. The `hours < 24` guard is always true when `days === 0`.

3. **Consider pagination** - The feed currently fetches up to `DEFAULT_LIST_LIMIT` (50) promotions. As the promotion database grows, adding pagination (offset/cursor + "Load More" or infinite scroll) would improve performance and UX. This is a future enhancement, not a blocker.
