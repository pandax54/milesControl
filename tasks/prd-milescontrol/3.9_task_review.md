### 🟢 Minor Issues

1. **`src/components/promotions/promotion-feed.tsx:51`** — The `hasFilters` check for the empty state only inspects `type` and `programId` but not `status`. If a user explicitly selects "Active" status, the empty state says "Try adjusting your filters" even though status is technically a filter. The behavior is reasonable since Active is the default, but slightly inconsistent. Non-blocking.

2. **Future consideration: pagination** — The feed currently fetches up to `DEFAULT_LIST_LIMIT` (50) promotions. As the promotion database grows, adding pagination (offset/cursor + "Load More") would improve performance and UX. Not a blocker for this task.

## ✅ Positive Highlights

1. **Clean separation of pure deadline logic** — `computeTimeRemaining` and `formatTimeRemaining` are extracted into `deadline-utils.ts` as pure functions with thorough tests (7 compute + 5 format scenarios), making the countdown logic fully testable without React rendering.

2. **Comprehensive Zod validation schema** — `promotion-feed.schema.ts` provides clean, reusable filter definitions with companion label records (`PROMO_TYPE_LABELS`, `PROMO_SORT_LABELS`) that centralize display strings alongside schema types, avoiding scattered label definitions across components.

3. **Excellent test coverage** — 17 schema tests, 8 action tests, 12 countdown tests, and 12 new service tests (listPromotions, listPromotionPrograms, getPromotionById, PromotionNotFoundError). All new code at 100% coverage.

4. **Smart sort ordering with tiebreaker** — `buildSortOrder` in the service adds a secondary `detectedAt desc` sort for non-detectedAt sorts, ensuring stable, predictable ordering.

5. **Embedded calculator integration** — `PromotionCard` pre-fills calculator defaults from promotion data (`bonusPercent`, `purchasePricePerK`, `clubExtraBonus`), fulfilling the PRD requirement for "one-click cost calculator" embedded in promo cards (F3.5).

6. **Proper Pino logging** — Server actions use `logger.error({ err: error }, 'message')` with structured context. No `console.log` anywhere in the new code.

7. **Correct `programId` filter** — The service uses `OR: [{ sourceProgramId }, { destProgramId }]` to match programs on either side of a promotion, which is the correct behavior for the filter bar's "Program" dropdown.

8. **Thoughtful UX details** — Context-aware empty state (different messaging for "no results with filters" vs "no promotions at all"), loading state with disabled controls via `useTransition`, "Clear filters" button only shown when filters are active, countdown urgency styling (red when <1 day).

9. **`listPromotionPrograms` query is efficient** — Only returns programs that actually appear in promotions (via `{ some: {} }` relation filter), avoiding showing irrelevant programs in the filter dropdown.

10. **Previous review issues fully addressed** — Magic number extracted to `DEFAULT_PURCHASE_PRICE_PER_POINT`, runtime rating validation added via `PROMOTION_RATINGS.includes()` check, filter handler refactored to discriminated `switch`, and `useMemo` added for deadline timestamp dependency.

11. **No `any`, no `var`, no `console.log`, no direct `process.env`** — All code follows project strict TypeScript and quality rules.
