### 🟢 Minor Issues

1. **Magic number for default purchase price** — `src/components/promotions/promotion-card.tsx:38`

   The value `0.028` (R$28/k expressed as per-point price) is a magic number. While it's only used as a calculator default, it would benefit from being a named constant for clarity.

   ```ts
   const DEFAULT_PURCHASE_PRICE_PER_POINT = 0.028; // R$28 per milheiro
   ```

2. **Type assertion for rating field** — `src/components/promotions/promotion-card.tsx:57`

   `const rating = promotion.rating as PromotionRating | null;` casts the DB string to a union type without validation. This is safe as long as the DB only stores valid enum values, but a runtime check (e.g., checking membership in the rating values array) would be more defensive.

3. **Type assertion in filter handler** — `src/components/promotions/promotion-filters.tsx:35`

   `(updated as Record<string, string>)[key] = value;` bypasses TypeScript's property access checks. The risk is contained (keys are constrained by the union type in the function signature), but a discriminated update approach would be slightly more type-safe.

4. **`useEffect` dependency uses `.getTime()` expression** — `src/components/promotions/deadline-countdown.tsx:25`

   The dependency `[deadlineDate.getTime()]` is a computed expression inside the dependency array. While functionally correct (avoids unnecessary re-runs when Date reference changes but value stays the same), ESLint's `react-hooks/exhaustive-deps` may flag this. A memoized timestamp value would be more idiomatic. This is cosmetic and does not affect correctness.

## Recommendations

1. **Extract magic number** — Replace `0.028` in `buildCalculatorDefaults` with a named constant (`DEFAULT_PURCHASE_PRICE_PER_POINT`) for clarity.

2. **Consider pagination** — The feed currently fetches up to `DEFAULT_LIST_LIMIT` (50) promotions. As the promotion database grows, adding pagination (offset/cursor + "Load More" or infinite scroll) would improve performance and UX. This is a future enhancement, not a blocker.

3. **Add runtime guard for rating** — Consider validating `promotion.rating` against the known rating values before using it for badge variant lookup, to handle any unexpected DB values gracefully.
