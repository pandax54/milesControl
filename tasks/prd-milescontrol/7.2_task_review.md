### 🟢 Minor Issues

**2. `.then()` used inside `useEffect` for service worker registration**  
`src/components/pwa/dashboard-pwa-card.tsx` — lines 99–110

The project rules (node.md) state: *"Use `async/await`; do not mix with `.then()`."* The service worker registration uses `.then()` with two callbacks:

```ts
// current
navigator.serviceWorker.register('/sw.js').then(
  () => { if (isActive) setServiceWorkerStatus('ready'); },
  () => { if (isActive) setServiceWorkerStatus('error'); },
);
```

**Suggested fix:**

```ts
// preferred
(async () => {
  try {
    await navigator.serviceWorker.register('/sw.js');
    if (isActive) setServiceWorkerStatus('ready');
  } catch {
    if (isActive) setServiceWorkerStatus('error');
  }
})();
```

---

**3. `parseDashboardOfflineSnapshot` exported but unused in production code**  
`src/lib/pwa/dashboard-offline-snapshot.ts` — line 131

The function is only consumed in tests. The `offline.html` page reads and parses localStorage directly via `JSON.parse()` (a deliberate choice since it is a self-contained vanilla-JS page). If the function is intended for future use (e.g., a future React offline UI), retaining the export is fine — add a comment explaining the intent. If it will never be consumed by app code, consider marking it `@internal` or removing it to avoid misleading a future developer.

---

**4. `innerHTML` with unsanitised snapshot data in `offline.html`**  
`public/offline.html` — lines 302–333

The offline page injects string fields from a localStorage object directly into `innerHTML` template literals (e.g., `enrollment.program.name`, `transfer.sourceProgramName`). localStorage is origin-scoped, so exploitation requires prior XSS on the same origin — the risk is low but not zero.

**Suggested fix:** Use `textContent` assignment for dynamic values, or apply a minimal HTML-escape helper:

```js
function escapeHtml(str) {
  return String(str)
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;');
}
// then: escapeHtml(enrollment.program.name) inside template literals
```

---

**5. Service worker is plain JavaScript, not TypeScript**  
`public/sw.js`

The project rules require all source code to be TypeScript with strict mode. The service worker lives in `public/` and is not processed by the Next.js bundler, which is why plain JS is used. This is a known limitation of the current setup (no `next-pwa` or similar build plugin). No action required unless the team adopts a SW build pipeline, but worth noting for future work.

## ✅ Positive Highlights

- **Solid cache versioning**: `CACHE_VERSION = 'v1'` constant with deterministic cache names; activate handler clears all stale `milescontrol-*` caches — no cache leaks across versions.
- **Correct stale-while-revalidate strategy**: serves cached assets immediately while revalidating in the background; clones responses before storing.
- **Robust offline navigation fallback**: navigations get the network first, then the cached `offline.html` as the fallback, never throwing.
- **Proper component cleanup**: every `useEffect` in `DashboardPwaCard` returns a cleanup function; the `isActive` flag prevents state updates after unmount.
- **Zod validation for the snapshot schema**: `parseDashboardOfflineSnapshot` uses Zod's `safeParse` so invalid stored data never propagates to the UI.
- **Correct Next.js manifest integration**: `src/app/manifest.ts` uses the `MetadataRoute.Manifest` type and is type-safe; no manual route wiring needed.
- **iOS install instructions**: iOS Safari does not fire `beforeinstallprompt`, so the component correctly provides contextual instructions for iPhone users.
- **`viewport` export separated from `metadata`**: follows Next.js 14+ best practice of exporting `viewport` separately to avoid the deprecation warning.
- **Correct HTTP headers in `next.config.ts`**: `Service-Worker-Allowed: /` header enables the SW to control the full origin scope; `no-cache` on the SW file prevents stale SW deployments.
