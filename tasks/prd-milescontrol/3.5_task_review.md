### 🟡 Major Issues

**1. No error handling in `storePromotions` loop -- a single failing promotion aborts the entire batch**

File: `src/lib/services/promotion.service.ts`, lines 113-140

The `for` loop in `storePromotions` has no try/catch. If any Prisma operation fails for one promotion (e.g., a unique constraint race condition, a transient database error), the entire batch is aborted and the function throws, losing the progress of all previously processed promotions in the batch. Callers receive no partial result.

Suggested fix: Wrap each iteration in a try/catch that logs the error and increments a `failed` counter, then include `failed` in the result. Alternatively, if atomicity is desired, wrap the entire loop in a Prisma `$transaction`.

```typescript
// Option A: Resilient per-item processing
let failed = 0;
for (const scraped of promotions) {
  try {
    // ... existing logic ...
  } catch (err) {
    logger.error({ err, sourceUrl: scraped.sourceUrl }, 'Failed to store promotion');
    failed++;
  }
}
```

**2. `rawContent: scraped.rawContent ?? null` uses nullish coalescing on a non-optional field**

File: `src/lib/services/promotion.service.ts`, lines 176 and 201

The `ScrapedPromotion` interface declares `rawContent` as `readonly rawContent: string` (required, non-optional). Using `?? null` on a required `string` field is misleading -- it suggests the field could be undefined/null when the type guarantees it will always be present. While this is not a runtime bug, it masks a type mismatch between the interface and the database column (`rawContent String?` in the schema is nullable but the scraped type is not optional).

Suggested fix: Either make the `rawContent` field optional in `ScrapedPromotion` (`readonly rawContent?: string`) if scrapers may not always provide it, or simply use `scraped.rawContent` directly without `?? null` to keep the code consistent with the type definition.

## Recommendations

1. **Add per-item error handling in `storePromotions`**: Wrap each loop iteration in try/catch to prevent a single failure from aborting the entire batch. Add a `failed` counter to `StorePromotionsResult`. This is the most impactful improvement.

2. **Resolve the `rawContent` type inconsistency**: Either make `rawContent` optional in `ScrapedPromotion` or remove the `?? null` fallback in the service to keep types and runtime behavior aligned.

3. **Consider adding a test for Prisma errors during storage**: Verify that if `prisma.promotion.create` throws, the function behaves as expected (currently it would propagate the error and lose partial progress).

4. **Add a brief "why" comment on `PromotionNotFoundError`**: Document that it is intended for use in the API/controller layer in a future task.

5. **Run `pnpm prisma db push`** to apply the `@unique` constraint on `sourceUrl` to the development database before merging.

## Verdict

The implementation is solid, well-tested, and follows project standards closely. The deduplication logic is thoughtfully designed with appropriate same-source and cross-source strategies. The one major observation is the lack of error handling within the `storePromotions` batch loop, which could cause partial data loss if a single promotion fails to persist. This is a non-blocking concern for a development-phase service that will be called in controlled scraper contexts, but should be addressed before production use. The `rawContent` type inconsistency is a minor papercut. Overall, the code is approved for merge with the noted observations.
