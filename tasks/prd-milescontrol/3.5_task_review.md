## Summary

Task 3.5 implements the promotion deduplication and storage service that bridges scraped promotions (tasks 3.0-3.4) into the database. The service resolves scraped program names to database IDs via case-insensitive lookup, deduplicates promotions via same-source URL matching (backed by a `@unique` schema constraint) and cross-source heuristic matching (type + programs + bonus/discount within a 30-day window), computes ACTIVE/EXPIRED status based on deadlines, and provides query functions for listing and retrieving promotions. Previous review findings (missing per-item error handling, rawContent nullish coalescing, PromotionNotFoundError comment) have all been addressed. The implementation is clean, well-structured, and thoroughly tested with 33 tests. One non-blocking observation remains regarding the unhandled ScraperRun update error path.

### ЁЯЯб Major Issues

**1. Unhandled error on ScraperRun update may discard storage results**

File: `src/lib/services/promotion.service.ts`, lines 153-158

The `scraperRunId` update at lines 153-158 runs after the batch loop completes but before the result is returned. If `prisma.scraperRun.update()` fails (e.g., invalid `scraperRunId`, transient DB error), the function throws and the caller never receives the `StorePromotionsResult` тАФ even though all promotions were successfully stored. This is a non-critical side-effect that should not prevent reporting storage results.

Suggested fix:
```typescript
if (scraperRunId) {
  try {
    await prisma.scraperRun.update({
      where: { id: scraperRunId },
      data: { newPromos: created },
    });
  } catch (error) {
    logger.error(
      { err: error, scraperRunId, created },
      'Failed to update ScraperRun with new promo count',
    );
  }
}
```

Add a corresponding test that verifies the function still returns results when `scraperRun.update` rejects.

## тЬЕ Positive Highlights

1. **Clean two-tier deduplication strategy**: Same-source URL match updates existing records; cross-source heuristic match (type + programs + bonus/discount within 30-day window) correctly skips duplicates from different blogs covering the same promotion.
2. **Schema-level integrity**: `@unique` on `sourceUrl` enforces dedup at the database level, correctly backing the `findUnique` call in `findBySourceUrl`.
3. **Per-item error handling**: Each promotion in the batch is wrapped in try/catch with a `failed` counter (previous review fix), ensuring one bad promotion doesn't abort the batch.
4. **Comprehensive test coverage**: 33 tests cover all exported functions with edge cases (empty arrays, null programs, unresolvable names, mixed batch outcomes, error paths, deadline status variations, optional field nullification).

8. **`readonly` on interfaces**: `StorePromotionsResult` properties are all `readonly`; input uses `readonly ScrapedPromotion[]`.
9. **Case-insensitive program resolution**: `mode: 'insensitive'` in `resolveProgramId` handles inconsistent casing from different scraper sources.
10. **Well-designed query functions**: `listPromotions` supports optional filtering with sensible defaults; `getPromotionById` includes related programs.
