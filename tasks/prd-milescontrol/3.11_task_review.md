## Summary

Task 3.11 implements the Vercel Cron job that runs all four promotional scrapers every 30 minutes. The implementation delivers:

- `GET /api/cron/scrape-promos` route handler with CRON_SECRET bearer authentication and a `maxDuration = 300` export for Vercel's 5-minute function limit
- `scrape-promos.service.ts` orchestrating all four scrapers sequentially, skipping any with 3+ consecutive failures, and marking expired promotions after the run
- `scraper-run.service.ts` providing create/update/query helpers for the `ScraperRun` Prisma model, including `countConsecutiveFailures`
- A public `scraperName` getter on `BaseScraper` that cleanly exposes the protected `name` field without bracket-notation hacks
- `vercel.json` with both the `*/30 * * * *` `scrape-promos` entry and the pre-existing `balance-digest` weekly entry

A follow-up fix commit (`dc38003`) resolved the two minor issues identified in the initial review: error responses are now aligned with the project envelope `{ error: { code, message } }`, and the previously misleading test description was corrected.

All 951 tests pass, TypeScript strict mode is clean, and overall coverage is 98.46% — well above the 80% threshold. Both new service files achieve 100% statement, branch, function, and line coverage.

## Positive Highlights

- Security is correctly handled: `CRON_SECRET` is read from the Zod-validated `env` object (never `process.env` directly), and `verifyCronSecret` cleanly handles three cases — secret set and matching, secret set and mismatched, and secret unset with dev/prod differentiation.
- The `maxDuration = 300` export correctly signals to Vercel that this route may run up to 5 minutes. The inline comment explains the rationale.
- The `scraperName` public getter on `BaseScraper` is the clean solution to exposing the protected `name` field — no bracket-notation hacks or type casts needed downstream.
- The orchestration service degrades gracefully at every failure boundary: per-scraper failures are captured without aborting the run, `markExpiredPromotions` failure is caught with a structured error log and returns `expiredCount: 0`, and `countConsecutiveFailures` failure causes the scraper to proceed rather than be blocked.
- All logging uses Pino with data-first syntax and `err` as the error property key, fully compliant with the logging standard. No `console.log` anywhere in the new files.
- The `vercel.json` cron schedule `*/30 * * * *` is correct and the pre-existing `balance-digest` entry is preserved.
- Test suite is comprehensive: 7 route handler tests cover all auth permutations plus the error-mapping path; 11 service tests cover skip logic, failure aggregation, expiration errors, empty-promotions guard, `scraperRunId` passthrough, duration measurement, and boundary conditions at exactly-at-threshold and strictly-above-threshold values; 13 scraper-run service tests cover all CRUD functions and the `countConsecutiveFailures` counting logic.
- AAA pattern is consistently followed and all tests are fully independent.
- Both `scrape-promos.service.ts` and `scraper-run.service.ts` are well within the 300-line class / 50-line function limits.
- `readonly` is applied consistently across all interface properties and the `CompleteScraperRunInput` input object.
- Import ordering follows the project standard: external libraries first, then `@/` internal imports, then relative imports.
