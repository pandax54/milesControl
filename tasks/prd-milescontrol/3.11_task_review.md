The two issues identified in the initial review cycle were resolved in the fix commits:

1. **Error response envelope** — `scrape-promos/route.ts` (lines 10, 36) and `balance-digest/route.ts` now return `{ error: { code: string, message: string } }`, matching the project-wide convention.
2. **Test description accuracy** — The test `'should skip scraper with more than 3 consecutive failures'` now uses mock value `4` (strictly above the threshold), making the description and mock data consistent.

One observation that is informational rather than blocking: `countConsecutiveFailures` in `scraper-run.service.ts` fetches only the last 3 records (capped by `take: CONSECUTIVE_FAILURE_THRESHOLD`). In practice this means the function can never return a value greater than 3. The skip logic in `scrape-promos.service.ts` is correct for its purpose (`failures >= 3` still fires), and the skip reason message `${failures} consecutive failures` will always show "3 consecutive failures" regardless of actual depth. This was a deliberate design choice from task 3.0 and does not affect correctness — the skip is triggered appropriately. No change required.

## Positive Highlights

- Security is correctly handled: `CRON_SECRET` is read from the Zod-validated `env` object (never `process.env` directly). The `verifyCronSecret` function cleanly handles three cases: secret set and matching, secret set and mismatched, and secret unset with dev/prod differentiation via the `IS_DEVELOPMENT` flag.
- The `maxDuration = 300` export on the route signals to Vercel that this function may run up to 5 minutes, appropriate for sequential scraper runs. The inline comment explains the rationale.
- The `scraperName` public getter on `BaseScraper` is the clean solution to exposing the protected `name` field — no bracket-notation hacks or type casts needed downstream in the orchestration service.
- The orchestration service degrades gracefully at every failure boundary: per-scraper failures are captured as `error` fields without aborting the overall run; `markExpiredPromotions` failure is caught and logged with a structured error key, returning `expiredCount: 0`; and `countConsecutiveFailures` failure causes the scraper to proceed rather than be blocked unnecessarily.
- All logging uses Pino with data-first syntax and `err` as the error property key (`logger.error({ err: error, scraperName }, '...')`), fully compliant with the logging standard. No `console.log` anywhere in the new files.
- The `vercel.json` cron schedule `*/30 * * * *` is correct. The pre-existing `balance-digest` weekly entry is preserved without regression.
- The response shape follows the project envelope: `{ data: { scrapers, totalCreated, totalUpdated, totalFailed, expiredCount, durationMs } }` on success; `{ error: { code, message } }` on failure.
- Test suite is comprehensive: 7 route handler tests cover all auth permutations, the dev-mode bypass, the production-reject path, success path with full result mapping, and the 500 error path; 11 service tests cover skip logic (at threshold, above threshold, below threshold), failure aggregation across scrapers, expiration errors, empty-promotions guard, `scraperRunId` passthrough, duration measurement, and `countConsecutiveFailures` throwing.
- AAA pattern is consistently followed, tests are fully independent, and `beforeEach` resets all mocks.
- Both `scrape-promos.service.ts` (139 lines) and `scraper-run.service.ts` (93 lines) are well within the 300-line class / 50-line function limits.
- `readonly` is applied consistently across all interface properties (`ScraperRunResult`, `ScrapePromosResult`, `CompleteScraperRunInput`).
- Import ordering follows the project standard: external libraries first, then `@/` internal imports, then relative imports.
- Named constants are used throughout: `CONSECUTIVE_FAILURE_SKIP_THRESHOLD` in `scrape-promos.service.ts` and `CONSECUTIVE_FAILURE_THRESHOLD` in `scraper-run.service.ts`. No magic numbers.
