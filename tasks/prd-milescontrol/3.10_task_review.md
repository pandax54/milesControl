## Summary

Task 3.10 implements personalized promo matching — highlighting promotions relevant to the user's enrolled programs with contextual messages like "Relevant for you: You have 15.000 Livelo points — transfer with 90% bonus." The implementation adds a pure matching service (`promo-matcher.service.ts`) with 16 unit tests across two `describe` blocks, a `PersonalizedBadge` presentational component, and integrates matching into the existing promotion feed and card components. The server-side promotions page fetches user enrollments in parallel via `Promise.all`, maps them to a clean DTO (`EnrollmentSummary`), and passes them to the client component which memoizes matching with `useMemo`. The architecture is clean: pure functions with zero external dependencies, O(1) Map lookup per card, typed interfaces with `readonly` properties, no `any` types, no `console.log`, and no magic numbers.

### 🟢 Minor Issues

1. **Unreachable `return null` at `promo-matcher.service.ts:83`** — The final `return null` after the `sourceEnrollment` and `destEnrollment` if-blocks is logically unreachable because line 52-54 already returns null when both are falsy. It serves as a TypeScript safety guard and accounts for the 96.96% statement coverage rather than 100%. Acceptable — no action needed.

2. **Map recreated per promotion in batch matching** — `matchPromotion` allocates a new `enrollmentByProgramId` Map (lines 40-42) on each call. When called in a loop via `matchPromotions` (line 101), this creates N maps for N promotions with the same enrollment data. A micro-optimization would be to build the map once in `matchPromotions` and pass it to an internal helper. For typical data volumes (tens of promotions, single-digit enrollments), performance impact is negligible. No action required.

3. **`EMPTY_ENROLLMENTS` constant prevents React invalidation but is module-scoped** — In `promotion-feed.tsx:26`, the `EMPTY_ENROLLMENTS` array is declared at module scope to prevent `useMemo` invalidation when no enrollments are passed. This is a correct optimization pattern. The only minor note is that if the parent always provides enrollments (which the promotions page does), the fallback is unused. Completely fine — defensive design.

## ✅ Positive Highlights

1. **Pure function architecture** — The matching service has zero dependencies on databases, APIs, or global state. It accepts typed inputs and returns typed outputs, making it trivially testable and reusable across both server (for admin promo-client matching in Phase 6) and client contexts.

2. **Named constants for relevance scores** — `SOURCE_MATCH_SCORE` (10), `DESTINATION_MATCH_SCORE` (5), and `BOTH_MATCH_SCORE` (15) replace magic numbers and document the scoring hierarchy.

3. **Comprehensive test coverage** — 16 tests across two `describe` blocks covering all match types (SOURCE, DESTINATION, BOTH), edge cases (empty enrollments, null programs, zero balance, null `bonusPercent` on TRANSFER_BONUS type, promotion with only destination program), batch operations (multiple matches, no matches, empty list), and O(1) Map lookup verification. Each test follows the AAA pattern with descriptive names.

4. **Efficient data structure** — `matchPromotions` returns `Map<string, PromoMatch>` for O(1) per-card lookup in the rendering loop, avoiding repeated linear scans over the match list.

5. **`useMemo` with correct dependencies** — Match computation is memoized on `[promotions, enrollments]`, correctly recomputing when filter changes trigger a promotions refresh while avoiding recomputation on unrelated re-renders.

6. **Clean type design** — `EnrollmentSummary` decouples the matcher from Prisma entity types. All interface properties are `readonly`. `PromoMatchType` uses a union type (`'SOURCE' | 'DESTINATION' | 'BOTH'`) rather than an enum, following project conventions.

7. **Contextual, user-facing reason strings** — Include the user's actual balance formatted for Brazilian locale (`pt-BR`, e.g., "15.000") and bonus percentage where applicable, directly matching the PRD requirement: "Relevant for you: you have 15,000 Livelo points."

8. **Minimal presentational component** — `PersonalizedBadge` is a thin, stateless component with a typed icon mapping constant (`MATCH_ICON`) and zero side effects. Clean separation between matching logic and rendering.

9. **Server-side parallel fetching** — The promotions page fetches promotions, programs, and enrollments concurrently with `Promise.all`, avoiding a three-step waterfall. The DTO mapping (`rawEnrollments` → `EnrollmentSummary[]`) happens server-side, keeping the client payload minimal.

10. **Clean serialization boundary** — `EnrollmentSummary` contains only primitives (`string`, `string`, `number`), ensuring clean serialization over the RSC boundary from server to client component.

11. **Consistent import order** — All files follow external → internal (`@/`) → relative convention.
