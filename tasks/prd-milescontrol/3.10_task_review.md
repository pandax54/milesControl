## Summary

Task 3.10 implements the "Personalized promo matching" feature described in the PRD: highlighting promotions relevant to the user's enrolled programs with contextual reasons like "You have 15,000 Livelo points -- transfer with 90% bonus." The implementation adds a pure matching service (`promo-matcher.service.ts`) with 16 comprehensive unit tests, a presentational badge component (`PersonalizedBadge`), and wires the matching into the existing promotion feed page and card components. Code quality is high with no `any` types, no `console.log`, no magic numbers, and full adherence to project conventions. Two minor observations noted below.

### Minor Issues

**1. Non-null assertion on `destEnrollment` (promo-matcher.service.ts, line 77)**

The code uses `destEnrollment!` as a non-null assertion. While the logic is correct (the prior branches guarantee `destEnrollment` is defined at that point), an explicit guard or restructuring would be more defensive and idiomatic TypeScript:

```typescript
// Current (line 77):
reason: buildDestinationMatchReason(destEnrollment!),

// Suggested alternative — early return makes the non-null assertion unnecessary:
if (destEnrollment) {
  return {
    promotionId: promotion.id,
    matchType: 'DESTINATION',
    reason: buildDestinationMatchReason(destEnrollment),
    relevanceScore: DESTINATION_MATCH_SCORE,
  };
}

return null;
```

This eliminates the `!` operator entirely and makes the control flow self-documenting. Low priority since the current code is logically sound.

**2. Default parameter `enrollments = []` creates new reference on each render (promotion-feed.tsx, line 25)**

```typescript
export function PromotionFeed({ initialPromotions, programs, enrollments = [] }: PromotionFeedProps)
```

The default `[]` creates a new array reference on every render, which would cause the `useMemo` on line 30-33 to recompute unnecessarily if a parent omits the `enrollments` prop. Currently the only consumer (`PromotionsPage`) always passes `enrollments` explicitly, so this has no runtime impact. If other consumers are added in the future, consider extracting the default to a module-level constant:

```typescript
const EMPTY_ENROLLMENTS: EnrollmentSummary[] = [];

export function PromotionFeed({ initialPromotions, programs, enrollments = EMPTY_ENROLLMENTS }: PromotionFeedProps)
```

## Positive Highlights

1. **Pure function architecture**: The matching service has zero dependencies on databases, APIs, or global state. It accepts typed inputs and returns typed outputs, making it trivially testable and reusable across server and client contexts.

2. **Named constants for relevance scores**: `SOURCE_MATCH_SCORE`, `DESTINATION_MATCH_SCORE`, and `BOTH_MATCH_SCORE` replace what could have been magic numbers. The scoring model is clear and easily adjustable.

3. **Comprehensive test coverage**: 16 tests covering all match types (SOURCE, DESTINATION, BOTH), edge cases (empty enrollments, null programs, zero balance, null bonusPercent on TRANSFER_BONUS), and batch operations. Each test follows the AAA pattern with clear, descriptive names.

4. **Efficient data structure choice**: `matchPromotions` returns a `Map<string, PromoMatch>` for O(1) per-card lookup, avoiding repeated linear scans in the rendering loop.

5. **`useMemo` integration**: The match computation is memoized with correct dependencies (`promotions` and `enrollments`), avoiding unnecessary recomputation when only filters or unrelated state changes.

6. **Clean type design**: `EnrollmentSummary`, `PromoMatch`, and `PromoMatchType` are well-defined with `readonly` properties. The `EnrollmentSummary` acts as a lightweight DTO that decouples the matcher from Prisma's enrollment model.

7. **Contextual, user-facing reasons**: The reason strings include the user's actual balance and the bonus percentage where applicable (e.g., "You have 15,000 Livelo points -- transfer with 90% bonus"), matching the PRD requirement precisely.

8. **Minimal, focused UI component**: `PersonalizedBadge` is a thin presentational component with no state, no side effects, and clear icon mapping via a typed constant.

9. **Server-side enrollment fetch with `Promise.all`**: The promotions page fetches promotions, programs, and enrollments in parallel, avoiding a waterfall. The mapping to `EnrollmentSummary` is done server-side before serialization.

10. **Import organization**: All files follow the external-then-internal-then-relative import order convention consistently.
