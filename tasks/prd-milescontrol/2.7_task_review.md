## Summary

Task 2.7 implements the potential balance calculator feature inspired by Oktoplus. ## Issues Found

### Critical Issues

No critical issues found.

### Major Issues

1. **Missing `totalWithBestBonus` field from techspec** — `potential-balance.service.ts`

   The techspec defines `PotentialBalance` with an optional `totalWithBestBonus?: number` field described as "if a transfer bonus is active." The current implementation does not account for active transfer bonuses at all. While the field is optional, this was called out as a differentiator in the techspec and the PRD references transfer bonuses as a core concept.

   **Suggested fix**: Consider adding a `totalWithBestBonus` field that checks for active `Promotion` records with transfer bonuses and calculates the boosted total. This could be deferred to a follow-up task if transfer bonus tracking is not yet in place, but should be tracked.

### Minor Issues

1. **`as never` type casts in test mocks** — `potential-balance.service.test.ts`, lines 139, 153, 173, 208, 237, 268, 291, 314, 329, 359, 371, 396

   The `buildEnrollment` helper returns a shape that does not fully match the Prisma `ProgramEnrollment & { program: Program }` type, so `as never` is used to bypass the type mismatch on `mockResolvedValue`. This is a common workaround but `as never` is a stronger escape hatch than necessary.

   **Suggested fix**: Either type the `buildEnrollment` return value more precisely to match the Prisma generated type, or use `as unknown as typeof mockFindMany extends (...args: infer A) => Promise<infer R> ? R : never` — or more practically, just use `as any` which is the conventional mock escape hatch in test files. Alternatively, define a proper mock type that includes only the fields the service actually uses.

2. **Techspec field naming divergence** — `potential-balance.service.ts`, line 13

   The techspec uses `targetProgram: string` but the implementation uses `targetProgramName: string` and adds `targetProgramCurrency: string`. The implementation is arguably better (more descriptive, includes currency), but it diverges from the spec. This is minor since the spec is a guideline, not a contract.

## Recommendations

1. **Track `totalWithBestBonus` as follow-up** — Create a task to integrate active transfer bonus promotions into the potential balance calculation. This would complete the techspec interface and add significant value for users.

2. **Consider error handling for the Prisma query** — The `calculatePotentialBalances` function does not wrap the database call in a try/catch. If the query fails, the error will propagate to the page and cause a 500. Consider adding error handling with a fallback empty array and an `logger.error` call, or ensure the page-level error boundary handles this gracefully.

3. **Improve mock types in tests** — Replace `as never` casts with a properly typed mock builder to keep tests type-safe and catch regressions if the Prisma schema changes.

## Verdict

The implementation is solid, well-tested, and follows project standards. The service logic is correct, the UI is clean, and integration into the programs page is properly done with parallel data fetching. The only notable gap is the missing `totalWithBestBonus` calculation from the techspec, which can be tracked as a follow-up. Code is production-ready.
