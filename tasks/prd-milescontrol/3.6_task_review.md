## Summary

## ✅ Positive Highlights

1. **Excellent functional decomposition**: Five small pure functions (`calculateTotalMiles`, `calculateTotalCost`, `computeCostPerMilheiro`, `rateByThreshold`, `calculateCostPerMilheiro`) each do exactly one thing, are individually testable, and compose cleanly. Textbook adherence to the "one action per function" rule.

2. **Named constants for all thresholds and standard values**: `RATING_THRESHOLD_EXCELLENT`, `RATING_THRESHOLD_GOOD`, `RATING_THRESHOLD_ACCEPTABLE`, `POINTS_PER_MILHEIRO`, `STANDARD_PRICE_PER_POINT`, and `STANDARD_QUANTITY` eliminate all magic numbers from the calculation logic. Constants are co-located at the top of the module.

3. **`readonly` on all interface properties**: `CostCalculation`, `ScenarioComparison`, and `PresetScenario` all use `readonly` properties, ensuring immutability of returned values. `compareScenarios` also accepts `readonly CalculatorInput[]` to prevent mutation of the input array.

4. **Custom error classes with descriptive messages**: `InsufficientScenariosError` and `TooManyScenariosError` include the actual count received, making debugging straightforward. They correctly set `this.name` for proper error identification.

5. **100% test coverage**: 67 tests cover all exported functions with edge cases — zero values, negative values, boundary thresholds, missing optional fields, identical scenarios, very high bonuses, and rounding behavior. After the fix, all four metrics are at 100%.

6. **Thorough Zod validation schemas**: `calculatorInputSchema` validates non-negative prices, integer-only quantities, and applies a sensible default (`transferBonusPercent` defaults to 0). `compareInputSchema` constrains array length to 2-3 scenarios. Schema tests cover 14 validation paths including boundary values.

7. **Proper Pino logging**: Uses `logger.debug` for scenario comparison results and unratable promotion types — appropriate log level for a pure calculation layer. Structured data-first, message-last syntax. No `console.log` anywhere.

8. **Proper logger mocking in tests**: Logger is mocked via `vi.mock('@/lib/logger')` at the top of the test file, preventing real log output during tests.

9. **Proper use of Prisma's `PromoType`**: `ratePromotion` imports `PromoType` from the generated Prisma client, keeping the type aligned with the database schema and avoiding drift.

10. **Preset scenarios with realistic Brazilian market data**: Five presets (Livelo→Smiles 90%/100%, Esfera→Smiles 80%, Livelo→Azul 70%, Clube Smiles + Livelo) reflect real-world acquisition strategies and prices, providing immediate value for the Calculator UI (task 3.7).

11. **Graceful handling of unratable promotion types**: `ratePromotion` returns `null` for CLUB_SIGNUP and MIXED types with a debug log explaining why, rather than throwing or returning a misleading value.

12. **Precise docstrings for boundary behavior**: The `rateByThreshold` docstring clearly specifies exclusive/inclusive boundaries (e.g., "EXCELLENT: < R$12/k (exclusive)", "AVOID: <= 0 or >= R$20/k"), removing ambiguity about edge cases.
