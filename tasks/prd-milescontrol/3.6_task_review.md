## Summary

Task 3.6 implements the cost calculator service as a set of pure functions for calculating cost-per-milheiro, rating promotions by market thresholds, and comparing acquisition scenarios side-by-side. The implementation covers all four PRD requirements (F3.1 input, F3.2 output, F3.3 presets, F3.4 comparison) with clean, well-tested code. The architecture is sound: small composable pure functions (`calculateTotalMiles`, `calculateTotalCost`, `computeCostPerMilheiro`, `rateByThreshold`) are composed into the main `calculateCostPerMilheiro` function, which is then reused by `ratePromotion` and `compareScenarios`. Five preset scenarios provide realistic Brazilian market examples. Custom error classes handle boundary conditions in scenario comparison. Tests are thorough with 67 total tests (14 schema + 53 service) and coverage exceeds 98% across all metrics.

### Minor Issues

**1. Duplicated `STANDARD_PRICE_PER_POINT` constant across two private functions**

File: `src/lib/services/cost-calculator.service.ts`, lines 204 and 233

The value `0.028` (standard Livelo price per point) is declared as a local `const STANDARD_PRICE_PER_POINT` inside both `rateTransferBonus` (line 204) and `ratePointPurchase` (line 233). Per the code-standards rule "no magic numbers; extract to named SCREAMING_SNAKE_CASE constants", this should be a single module-level constant alongside the other constants at the top of the file (lines 6-9).

Suggested fix:
```typescript
// Add at line 9, alongside existing module-level constants
const STANDARD_PRICE_PER_POINT = 0.028;
const STANDARD_QUANTITY = 10000;
```
Then remove the local declarations from both functions.

**2. `ratePromotion` uses an inline string union instead of the existing Prisma `PromoType`**

File: `src/lib/services/cost-calculator.service.ts`, line 177

The `type` parameter in `ratePromotion` is typed as `'TRANSFER_BONUS' | 'POINT_PURCHASE' | 'CLUB_SIGNUP' | 'MIXED'` — a hardcoded string union that exactly mirrors the `PromoType` enum already defined in the Prisma schema and used throughout the rest of the codebase (`promotion.service.ts`, `types.ts`, `promotion-helpers.ts`). This duplication creates a maintenance risk: if `PromoType` values change in the schema, this function's type signature would silently drift.

Suggested fix:
```typescript
import type { PromoType } from '@/generated/prisma/client';

export function ratePromotion(promotion: {
  type: PromoType;
  bonusPercent?: number | null;
  // ...rest
}): CostCalculation | null {
```

However, there is a valid design argument for keeping pure functions decoupled from Prisma. If this is intentional to keep the calculator layer dependency-free, it should be documented with a "why" comment explaining the deliberate decoupling decision.

**3. The `rateByThreshold` boundary behavior at exactly R$12, R$16, and R$20 may be inconsistent with the PRD description**

File: `src/lib/services/cost-calculator.service.ts`, lines 131-137

The PRD/CLAUDE.md describes thresholds as: "Excellent (<R$12), Good (R$12-16), Acceptable (R$16-20), Avoid (>R$20)". The implementation uses strict less-than for all boundaries (`< 12`, `< 16`, `< 20`), which means exactly R$20 is rated as "AVOID". This is arguably correct (R$20 is ">" R$20's lower boundary), but the docstring at line 129 states "ACCEPTABLE: R$16-20/k" which implies R$20 should be included in ACCEPTABLE. The test at line 133 asserts `rateByThreshold(20)` returns `'AVOID'`, confirming the current behavior is intentional and tested, but the docstring description could be more precise (e.g., "ACCEPTABLE: R$16 to <R$20" or "AVOID: >= R$20"). This is a documentation-only issue.

## Recommendations

1. **Extract `STANDARD_PRICE_PER_POINT` and `STANDARD_QUANTITY` to module-level constants** — Eliminates duplication between `rateTransferBonus` and `ratePointPurchase`, and makes it easier to update the baseline price if market conditions change.

2. **Consider using Prisma's `PromoType` or add a "why" comment** — Either import the existing type for consistency with the rest of the codebase, or add a brief comment explaining the intentional decoupling from Prisma for pure-function portability.

3. **Clarify the docstring for `rateByThreshold` boundary behavior** — Change "ACCEPTABLE: R$16-20/k" to "ACCEPTABLE: R$16 to <R$20/k" and "AVOID: >= R$20/k" to avoid ambiguity about the boundary values.

4. **Consider a test for `ratePromotion` with `purchasePricePerK` as a `Decimal`** — The Prisma schema defines `purchasePricePerK` as `Decimal(10,2)`. When reading from the database, Prisma returns a `Decimal` object (not a plain `number`). The current `ratePointPurchase` function uses `Number(promotion.purchasePricePerK)` at line 223, which handles this correctly, but adding a test with a `Decimal`-like input would verify the coercion path.
