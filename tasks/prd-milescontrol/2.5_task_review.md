### Minor Issues

1. **Duplicated `formatNumber` utility across 4 components**
   - Files: `dashboard-summary-cards.tsx` (line 11), `dashboard-balances.tsx` (line 21), `dashboard-projections.tsx` (line 8), `dashboard-transfers.tsx` (line 10)
   - The identical `formatNumber` function (`new Intl.NumberFormat('pt-BR').format(value)`) is defined in 4 separate files. Similarly, `formatCurrency` is duplicated in `dashboard-subscriptions.tsx` and `dashboard-transfers.tsx` (and also exists in other components from previous tasks).
   - **Suggestion**: Extract both into a shared utility at `src/lib/utils/format.ts` and import from there. This is a minor DRY concern; it does not block this task but should be addressed as a follow-up to avoid further drift.

2. **Magic numbers in `rateTransfer` function**
   - File: `src/components/dashboard/dashboard-transfers.tsx`, lines 27-30
   - The cost-per-milheiro rating thresholds (12, 16, 20) are inline magic numbers. The domain concept and these exact thresholds are documented in the CLAUDE.md (Excellent < R$12, Good R$12-16, Acceptable R$16-20, Avoid > R$20).
   - **Suggestion**: Extract to named constants (e.g., `EXCELLENT_THRESHOLD = 12`, `GOOD_THRESHOLD = 16`, `ACCEPTABLE_THRESHOLD = 20`) or better yet, import them from a shared domain constants file since this rating logic will likely be reused in task 3.6 (cost calculator service).

## Recommendations

1. **Extract shared formatting utilities**: Create `src/lib/utils/format.ts` with `formatNumber` and `formatCurrency` functions. This will reduce duplication across current and future dashboard components.
2. **Extract rating thresholds to shared constants**: The cost-per-milheiro rating thresholds (12/16/20) will be needed again in the cost calculator service (task 3.6). Define them once in a domain constants file.
3. **Consider error boundary for dashboard**: The current page does not handle the case where `fetchDashboardData` throws (e.g., database connectivity issue). A `try/catch` wrapping the fetch with a user-friendly error state, or a Next.js `error.tsx` boundary for the dashboard route, would improve resilience.
