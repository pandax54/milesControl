### Minor Issues

1. **`as never` casts in test mocks** — `dashboard.service.test.ts` uses `as never` 27 times when passing mock data to Prisma mocks. While pragmatic for working around Prisma's complex generated types, this weakens type safety: if a Prisma return type changes (field added/renamed), the test won't catch the mismatch at compile time. This is inconsistent with the typed `buildMockCard()` pattern established in the credit-card tests after 2.4 review fixes.

   **Suggestion**: Build typed mock factories that return the actual Prisma types with their includes, or use `as unknown as ProgramEnrollmentWithProgram` for more explicit casts.

2. **Rating thresholds will need extraction for task 3.6** — `dashboard-transfers.tsx:18-20` defines `COST_THRESHOLD_EXCELLENT = 12`, `COST_THRESHOLD_GOOD = 16`, `COST_THRESHOLD_ACCEPTABLE = 20` with a `rateTransfer()` function. These match the PRD domain thresholds (F2.4) and are already named constants (not magic numbers), which is good. However, this rating logic will be needed again in the cost calculator service (task 3.6) and promotion feed. Currently this is the only usage so no action needed now — when building task 3.6, extract to a shared `src/lib/utils/cost-rating.ts`.

3. **No unit tests for format utilities** — `src/lib/utils/format.ts` exports `formatNumber` and `formatCurrency` but has no co-located test file. These are simple `Intl.NumberFormat` wrappers so risk is low, but adding a few tests would match the project's testing standards.

## Positive Highlights

- **Parallel data fetching** — `fetchDashboardData` uses `Promise.all` for 4 independent queries (enrollments, subscriptions, transfers, projections), minimizing latency.
- **Decimal-to-number conversion at the service boundary** — `monthlyCost`, `totalCost`, and `costPerMilheiro` are all converted via `Number()` in the service layer before reaching UI. Consistent with the pattern from 2.4 review fixes.
- **Clean component decomposition** — 6 focused components, each with a single responsibility, clear props interface, and proper TypeScript typing. The page is a thin orchestrator.
- **Empty state handling** — Both the page level (no enrollments → onboarding CTA) and individual sections (no subscriptions, no transfers, no projections) have clear, helpful empty states with navigation links.
- **Staleness classification** — `classifyStaleness` is a pure, exported function with injected `now` parameter for testability. 6 boundary tests cover all edge cases including exact threshold values.
- **Readonly types throughout** — All interfaces use `readonly` properties and `readonly` arrays, enforcing immutability from service through to UI.
- **Dark mode support** — Staleness colors and alert banner include both light and dark mode classes (`dark:text-red-400`, `dark:bg-red-950`).
- **Shared format utilities** — `formatNumber` and `formatCurrency` are properly extracted to `src/lib/utils/format.ts` and imported consistently across all dashboard components.
- **Proper type exports** — Service exports types separately from functions, enabling clean imports in components.
- **Well-scoped Prisma queries** — Uses `select` for program fields, filters subscriptions to `status: 'ACTIVE'`, limits transfers to 5 with descending date order, uses `include` to avoid N+1.
- **Pino logging** — Structured log with relevant context (userId, counts) and descriptive message. No `console.log` anywhere.
