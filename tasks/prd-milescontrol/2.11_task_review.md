### 🟢 Minor Issues

**1. `as never` casts in service test**

- **File**: `src/lib/services/family-member.service.test.ts`, lines 112, 252, 315, 334, 350, 388
- **Description**: The `buildMockMember()` and `buildMockEnrollment()` factories are properly typed with Prisma types, but 6 `mockResolvedValue` calls still use `as never` to work around type mismatches between the plain type and Prisma's included-relation return shape. This is inconsistent with the pattern established in tasks 2.7/2.9/2.10 review fixes where typed factories eliminated all `as never` casts.
- **Suggested fix**: Create a type alias for the `findMany` / `create` return shape with included relations:
  ```typescript
  type MemberWithEnrollments = FamilyMember & {
    programEnrollments: (FamilyProgramEnrollment & {
      program: Pick<Program, 'id' | 'name' | 'type' | 'currency' | 'logoUrl' | 'website'>;
    })[];
  };
  ```
  Then use `buildMockMember()` returning `MemberWithEnrollments` for the `findMany` call, and similar for enrollment `create`/`update` calls.

**2. `as never` casts in actions test**

- **File**: `src/actions/family.test.ts`, lines 253, 343
- **Description**: `mockCreateEnrollment.mockResolvedValue({} as never)` and `mockUpdateEnrollment.mockResolvedValue({} as never)` use empty object casts. These should use a typed factory like `buildMockEnrollmentReturn()`, matching the pattern from `benefits.test.ts` (task 2.10 review fix).
- **Suggested fix**: Create a typed `buildMockEnrollmentReturn()` factory using `FamilyProgramEnrollment` from Prisma types.

**3. Duplicated `AvailableProgram` interface**

- **File**: `src/components/dashboard/family-member-card.tsx` (line 18) and `src/components/dashboard/family-enrollment-form-dialog.tsx` (line 25)
- **Description**: The `AvailableProgram` interface is defined identically in both files. Should be defined once and shared.
- **Suggested fix**: Export `AvailableProgram` from `family-member-card.tsx` (already exported) and import it in the enrollment form dialog instead of redefining it.

## ✅ Positive Highlights

1. **Strong ownership verification** — Every service operation verifies the family member belongs to the requesting user via `findFirst({ where: { id, userId } })` before mutation. Consistent and secure.

2. **Constrained relationship values** — Using `z.enum(RELATIONSHIPS)` in Zod schemas with shared `RELATIONSHIPS` constant and `RELATIONSHIP_LABELS` map. Defense in depth with test for invalid relationship rejection.

3. **Smart enrollment filtering** — `FamilyMemberCard` computes `programsForEnrollment` by excluding already-enrolled programs, preventing duplicate enrollment attempts in the UI.

4. **Balance staleness tracking** — `updateFamilyEnrollment` conditionally updates `balanceUpdatedAt` only when balance actually changes (`hasBalanceChange` check at line 164), matching the staleness indicator pattern from task 2.1.

5. **Shared action helpers** — Uses extracted `requireUserId`, `isAuthenticationError`, and `ActionResult` from `src/actions/helpers.ts`, maintaining consistency with tasks 2.10+.

6. **Edit dialog form reset on open** — Both edit dialogs reset form values from props when opened (`if (value) resetForm()`), preventing stale state across openings.

7. **Comprehensive test coverage** — 23 validator tests, 22 service tests, 25 action tests covering all success paths, error paths (not found, already exists, auth), and edge cases (null relationship, balance unchanged, null expirationDate).

8. **Parallel data fetch in page** — `Promise.all([listFamilyMembers, listPrograms])` for efficient server-side data loading.

9. **Staleness color constants** — `STALENESS_FRESH_DAYS` and `STALENESS_WARNING_DAYS` named constants in enrollment card instead of magic numbers.

10. **Pino logging** — All service operations log structured context (userId, familyMemberId, enrollmentId, programId) with clear messages. Error logging in actions uses the correct `{ err: error }` pattern.
