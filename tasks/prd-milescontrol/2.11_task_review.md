### Major Issues

**1. Relationship field accepts arbitrary strings instead of constrained values**

- **File**: `src/lib/validators/family.schema.ts`, lines 23 and 29
- **Description**: The `RELATIONSHIPS` array is defined as a typed constant with 5 allowed values (`spouse`, `child`, `parent`, `sibling`, `other`), and `RELATIONSHIP_LABELS` maps only those values. However, `createFamilyMemberSchema` and `updateFamilyMemberSchema` both define `relationship` as `z.string()` instead of `z.enum(RELATIONSHIPS)`. This means the schema will accept any string (e.g., `"xyzzy"`) as a valid relationship, bypassing the intended constraint. The UI uses a `<Select>` with only the known values, but the server-side validation is the actual security boundary.
- **Suggested fix**:
  ```typescript
  // In createFamilyMemberSchema:
  relationship: z.enum(RELATIONSHIPS).optional(),

  // In updateFamilyMemberSchema:
  relationship: z.enum(RELATIONSHIPS).nullable().optional(),
  ```

### Minor Issues

**1. Enrollment ordering uses `familyMemberId` instead of a meaningful sort**

- **File**: `src/lib/services/family-member.service.ts`, line 28
- **Description**: `orderBy: { familyMemberId: 'asc' }` sorts enrollments by the foreign key to the parent, which is the same value for all enrollments within a single family member. This effectively produces no meaningful ordering. Consider sorting by `createdAt: 'desc'` or `program.name: 'asc'` for a more useful display order.
- **Suggested fix**:
  ```typescript
  orderBy: { createdAt: 'desc' },
  ```

**2. `ProgramNotFoundError` is defined in the family service but may collide with other modules**

- **File**: `src/lib/services/family-member.service.ts`, lines 242-247
- **Description**: `ProgramNotFoundError` is a generic name. If a shared `ProgramNotFoundError` exists or is introduced elsewhere, having it defined locally here will create confusion. Consider either using a shared error class or naming it `FamilyProgramNotFoundError` to scope it clearly.

**3. `DialogTrigger` uses `render` prop pattern instead of `asChild`**

- **File**: `src/components/dashboard/family-member-form-dialog.tsx`, line 63 (and similar in other dialog files)
- **Description**: The dialogs use `<DialogTrigger render={<Button />}>` which is the Radix v2 API. This is consistent across the codebase so it is not a problem per se, but worth noting for awareness if the project migrates shadcn versions.

## Recommendations

1. **Constrain `relationship` field in Zod schemas** to use `z.enum(RELATIONSHIPS)` instead of `z.string()`. This is the most important fix -- the UI already constrains the input, but the server validation should be the authoritative boundary.
2. **Change enrollment ordering** from `familyMemberId: 'asc'` to `createdAt: 'desc'` or a program name sort for meaningful display order.
3. **Consider extracting `ProgramNotFoundError`** to a shared location if the same concept is needed across multiple services.

## Verdict

The implementation is solid and production-ready. It follows all established project patterns, has comprehensive test coverage, proper ownership verification, and correct logging. The relationship validation gap (major issue #1) is mitigated by the UI only offering valid options, but should be fixed server-side for defense in depth. The other issues are minor improvements. Approved with the recommendation to tighten the Zod enum constraint.
