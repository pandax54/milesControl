### 🟢 Minor Issues

1. **Remaining `as never` casts in service test** — `src/lib/services/family-member.service.test.ts` still has 6 `as never` casts on lines 154, 294, 357, 376, 392, and 430. These are on `mockResolvedValue` calls for Prisma operations that use `include` (returning `EnrollmentWithProgram`). The typed mock factories produce correct objects but the Prisma generic inference requires the cast. This is a known limitation, but for consistency with the `as never` cleanup documented in progress.txt for this task, consider centralizing a typed resolution helper.

2. **Duplicated `parseBalance` function** — `family-enrollment-form-dialog.tsx:52` and `edit-family-enrollment-dialog.tsx:60` define identical `parseBalance(value: string): number | null` functions. Following the established pattern of extracting shared form utilities (e.g., `transfer-form-utils.ts` from task 2.6), this should be consolidated into a shared module.

3. **Duplicated date formatting logic** — `edit-family-enrollment-dialog.tsx` uses `new Date(x).toISOString().split('T')[0]` in both initial state (line 44) and `resetForm` (line 54). A small `formatDateForInput` helper would reduce duplication.

## ✅ Positive Highlights

1. **Strong ownership verification** — Every service operation verifies the family member belongs to the requesting user via `findFirst({ where: { id, userId } })` before mutation, preventing cross-user access.

2. **Constrained relationship values** — `z.enum(RELATIONSHIPS)` with shared `RELATIONSHIPS` constant and `RELATIONSHIP_LABELS` map. Test for invalid relationship rejection included.

3. **Smart enrollment filtering** — `FamilyMemberCard` computes `programsForEnrollment` by excluding already-enrolled programs, preventing duplicate enrollment attempts at the UI level.

4. **Balance staleness tracking** — `updateFamilyEnrollment` conditionally updates `balanceUpdatedAt` only when balance actually changes, consistent with the staleness indicator system from task 2.1.

5. **Shared action helpers** — Uses `requireUserId`, `isAuthenticationError`, and `ActionResult` from `src/actions/helpers.ts`.

6. **Edit dialog form reset on open** — Both edit dialogs reset form values from props when opened, preventing stale state.

7. **Comprehensive test coverage** — 23 validator tests, 22 service tests, 25 action tests covering success paths, all 4 error types, auth failures, and edge cases (null relationship, unchanged balance, null expirationDate).

8. **Typed mock factories in action tests** — `buildMockEnrollmentReturn` and `buildMockMemberReturn` use proper Prisma types, matching the pattern from tasks 2.6–2.10.

9. **Shared `AvailableProgram` interface** — Imported from `family-member-card.tsx` rather than redefined in the enrollment form dialog.

10. **Pino logging** — All service operations log structured context (userId, familyMemberId, enrollmentId, programId) with the correct `{ err: error }` pattern for errors.

## Recommendations

2. **Address remaining `as never` casts** — The 6 casts in `family-member.service.test.ts` are a Prisma type inference limitation. Either document this as acceptable or centralize into a typed mock helper for consistency.

3. **Extract date formatting helper** — The `toISOString().split('T')[0]` pattern for date input formatting could be a small shared utility.
