## Summary

Task 7.1 implements the freemium gate layer that divides MilesControl into a Free tier and a Premium tier, following the Oktoplus model. The implementation introduces a clean, centralized access-control architecture covering:

- A static metadata module (`src/lib/freemium.ts`) declaring premium feature keys, descriptions, and tier feature lists.
- A service layer (`src/lib/services/freemium.service.ts`) with synchronous helpers and async Prisma-backed assertions.
- A `FreemiumTier` enum (`FREE | PREMIUM`) added to the Prisma schema and wired onto the `User` model with a `FREE` default.
- Server-side enforcement in all six premium-gated Server Actions (benefits, calculator/Redemption Advisor, explore, flights, alert Telegram channel, program enrollment).
- UI-level gating using a `PremiumFeatureCard` component and conditional rendering on every affected page.
- Telegram bot gating: `findUserByChatId` returns `null` for free-tier users so `/alerts`, `/promos`, and `/calc` commands respond with a premium upsell message.
- An `/upgrade` page that shows tier status, slot usage, and feature comparison.

The implementation is thorough, well-tested, and production-ready with 1564 tests passing, 94.48% statement coverage, zero TypeScript errors, and a clean production build.

### Minor Issues

**1. Silent gate bypass for unauthenticated callers in `computeRedemptionAdvisorAction` and `exploreDestinationsAction`**

File: `src/actions/calculator.ts` (lines 102-103), `src/actions/explore.ts` (lines 48-49)

When `session` is `null` (unauthenticated caller), the premium gate is skipped entirely and the action proceeds as if premium access has been granted. The conditional is `if (session?.user?.id) { await assertPremiumFeatureAccess(...) }`.

This is not a security hole in practice because both pages that invoke these actions redirect unauthenticated users to `/login` before the component can call the action, and the feature's personalized data step also short-circuits when there is no session. However, the behaviour is slightly inconsistent with the intent of the gate: a crafty caller invoking the Server Action directly without a session would bypass the check silently.

Suggested approach for a future hardening pass: either require authentication at the top of both actions (like the `requireUserId` helper used in `benefits.ts`) and return an auth error for unauthenticated callers, or add an explicit `else` branch returning `{ success: false, error: 'Authentication required' }` when there is no session.

**2. Indentation inconsistency in `alerts/page.tsx`**

File: `src/app/(alerts)/alerts/page.tsx` (lines 74-78, 99-103)

The JSX object spread inside the two `AlertConfigCard` calls uses inconsistent indentation — some inner properties are indented at two extra levels compared to the surrounding block. This does not affect behaviour but breaks the visual alignment pattern used throughout the rest of the codebase.

Example (lines 74-78):
```tsx
channels: config.channels as AlertChannelValue[],
programNames: config.programNames,
  promoTypes: config.promoTypes as PromoTypeValue[],   // extra indent
  minBonusPercent: config.minBonusPercent,              // extra indent
```

**3. Warning log in `sendTelegramAlerts` counts skipped-premium alerts as `failed`**

File: `src/lib/services/telegram-notification.service.ts` (lines 85-103)

When a TELEGRAM-channel match is found but the user is on the free tier, the alert config is filtered out of `chatIdByAlertConfigId` and the code falls through to the `if (!chatId)` branch, logging a warning "no chatId configured on alert config" and incrementing `failed`. This log message is misleading — the real reason the send was skipped is that the user is not a Premium subscriber, not that no chat ID was configured.

The distinction is observable in monitoring dashboards and could cause unnecessary alerts for operators investigating failed deliveries. A clearer log with a separate counter for premium-gated skips would be preferable, though this is cosmetic and low-priority.

## Positive Highlights

**1. Clean two-layer separation of gate logic**

The freemium module is split correctly: `src/lib/freemium.ts` contains pure static metadata (no imports, no async) while `src/lib/services/freemium.service.ts` owns all Prisma-backed assertions. This makes the metadata trivially tree-shakeable for the client bundle and keeps the service layer testable in isolation.

**2. Typed feature-key union prevents typos**

`PremiumFeatureKey` is derived from the `PREMIUM_FEATURE_KEYS` const array via `(typeof PREMIUM_FEATURE_KEYS)[number]`, so every call site passing a feature key gets compile-time checking. Adding or removing a feature automatically propagates through the type system.

**3. `assertProgramEnrollmentAvailable` skips the DB count for Premium users**

The short-circuit at line 100-102 of `freemium.service.ts` (`if (tier === 'PREMIUM') return;`) avoids a database round-trip for the fast path and is covered by a dedicated test that asserts `mockEnrollmentCount` was not called.

**4. Freemium gate is enforced at the Server Action layer, not just in the UI**

All six premium features gate at the action or service level, making the protection independent of the page rendering logic. A free-tier user who constructs a raw Server Action call (or whose client component renders unexpectedly) still cannot perform the gated operation.

**5. Consistent `PremiumFeatureCard` for hard-gated pages**

Pages where the entire feature is inaccessible (Explore, Benefits) render `PremiumFeatureCard` in place of the functional component, providing a uniform upgrade prompt. Pages where partial functionality is available (Flight Search, Calculator, Alerts) use inline conditional messaging so free users still derive value from the page.

**6. Telegram bot enforcement is defense-in-depth**

`findUserByChatId` filters on `freemiumTier === 'PREMIUM'` at the database query level, and `sendTelegramAlerts` additionally filters alert configs to `freemiumTier === 'PREMIUM'` before dispatching. The webhook command handlers then check the returned user for null and respond with a clear premium upsell message instead of silently ignoring the request.

**7. `/upgrade` page is contextually aware**

The page fetches `UserFreemiumAccessState` and renders differently for free vs premium users: free users see slot usage and an upgrade CTA; premium users see a confirmation that all features are unlocked. The upgrade page correctly does not expose any fake payment flow — it functions as an information page appropriate for a self-hosted product.

**8. Comprehensive test coverage for new service**

`freemium.service.test.ts` covers all seven exported functions including error classes (`PremiumFeatureRequiredError`, `ProgramEnrollmentLimitReachedError`), the DB short-circuit for premium users, remaining-slot calculation, and the error thrown when a user record is not found. Every test follows the AAA pattern with `vi.clearAllMocks()` in `beforeEach`.

**9. Schema migration is minimal and safe**

Adding `FreemiumTier` enum and `freemiumTier FreemiumTier @default(FREE)` to the User model is non-breaking: all existing rows get the `FREE` default, no data migration is required, and the change is backwards-compatible if the migration is rolled back.

**10. Pino logging throughout, no `console.log`**

A full-repository grep confirms zero `console.log` calls in the source tree. All new logging in the service and Telegram webhook follows the structured `{ context }, 'message'` pattern.

## Recommendations

1. **Future hardening (low priority)**: Standardise the unauthenticated path in `computeRedemptionAdvisorAction` and `exploreDestinationsAction` to return an explicit auth error rather than silently skipping the premium gate. This eliminates a theoretical gap if either action is called directly without a valid session outside of the normal page flow.

2. **Log message clarity (cosmetic)**: In `sendTelegramAlerts`, differentiate between "no chat ID on alert config" and "user is not Premium" when logging skipped deliveries. Consider a separate `premiumGated` counter in the result object so monitoring dashboards can distinguish configuration errors from intentional tier-based suppression.

3. **Indentation cleanup (cosmetic)**: Fix the off-by-two indentation inside the two `AlertConfigCard` JSX blocks in `src/app/(alerts)/alerts/page.tsx` to match the surrounding code style.
