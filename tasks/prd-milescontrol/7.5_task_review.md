### 🟡 Major Issues

**1. `analytics-identity.tsx` has no test file**  
File: `src/components/analytics/analytics-identity.tsx`  
The `AnalyticsIdentity` component (36 lines) contains non-trivial logic — dedup via `useRef`, conditional `name` spreading, and a `useEffect` that calls `identifyAnalyticsUser` exactly once per user switch. No `.test.tsx` exists for this file, meaning the identification dedup behavior and SSR/re-render safety are completely untested. The project rules require co-located tests and 80% coverage for all new code.

**Suggested fix:** Add `src/components/analytics/analytics-identity.test.tsx` covering:
- identifies user on mount
- does not re-identify when props re-render with the same `userId`
- re-identifies when `userId` changes

```tsx
// src/components/analytics/analytics-identity.test.tsx
import { render } from '@testing-library/react';
import { beforeEach, describe, expect, it, vi } from 'vitest';
import { AnalyticsIdentity } from './analytics-identity';

vi.mock('@/lib/analytics/client', () => ({
  identifyAnalyticsUser: vi.fn(),
}));

import { identifyAnalyticsUser } from '@/lib/analytics/client';
const mockIdentify = vi.mocked(identifyAnalyticsUser);

describe('AnalyticsIdentity', () => {
  beforeEach(() => vi.clearAllMocks());

  it('should identify the user on mount', () => {
    render(<AnalyticsIdentity userId="u1" email="a@b.com" role="USER" />);
    expect(mockIdentify).toHaveBeenCalledOnce();
    expect(mockIdentify).toHaveBeenCalledWith({ id: 'u1', email: 'a@b.com', role: 'USER' });
  });

  it('should not re-identify on unrelated re-renders with the same userId', () => {
    const { rerender } = render(<AnalyticsIdentity userId="u1" email="a@b.com" role="USER" />);
    rerender(<AnalyticsIdentity userId="u1" email="a@b.com" role="USER" />);
    expect(mockIdentify).toHaveBeenCalledOnce();
  });

  it('should re-identify when the userId changes', () => {
    const { rerender } = render(<AnalyticsIdentity userId="u1" email="a@b.com" role="USER" />);
    rerender(<AnalyticsIdentity userId="u2" email="b@c.com" role="ADMIN" />);
    expect(mockIdentify).toHaveBeenCalledTimes(2);
  });
});
```

### 🟢 Minor Issues

**3. Non-null assertion on optional env field**  
File: `src/lib/analytics/client.ts`, line 19  
`publicEnv.NEXT_PUBLIC_POSTHOG_KEY!` uses a `!` non-null assertion. It is safe in practice because the `if (!IS_ANALYTICS_ENABLED)` guard above guarantees the key is present when execution reaches `posthog.init`. However, the safety is an implicit contract between `IS_ANALYTICS_ENABLED` and the field's presence, not enforced by TypeScript. A narrowing approach would be safer.

**Suggested fix:**
```ts
const key = publicEnv.NEXT_PUBLIC_POSTHOG_KEY;
if (!IS_ANALYTICS_ENABLED || !key || ...) return;
posthog.init(key, { ... });
```

---

**4. Module-level mutable state in `client.ts`**  
File: `src/lib/analytics/client.ts`, line 12  
`let hasInitializedAnalytics = false` is module-level mutable state. In production this works correctly. In test environments it could lead to surprising test-ordering bugs if any test imports `client.ts` directly (rather than mocking it). All current tests mock the module, so this is not actively harmful, but it is a fragile pattern.

**Suggested fix (low priority):** Reset the flag in tests via `vi.resetModules()`, or refactor to a class/singleton that can be reset.

## ✅ Positive Highlights

- **Graceful opt-in**: Analytics is fully disabled (no errors, no network calls) when `NEXT_PUBLIC_POSTHOG_KEY` is not set — correct behaviour for local and CI environments.
- **Centralized env config**: `public-env.ts` uses a Zod schema with proper defaults and error messages instead of direct `process.env` access — exactly matches project convention.
- **Consistent SSR guards**: Every analytics function in `client.ts` checks `typeof window === 'undefined'` before any PostHog call, preventing server-side crashes.
- **Type-safe event registry**: `ANALYTICS_EVENTS as const` + derived `AnalyticsEventName` type ensures all event name strings are autocompleted and validated at compile time.
- **Identification dedup via `useRef`**: `AnalyticsIdentity` tracks the last identified `userId` in a ref, preventing redundant `identify` calls on every render without introducing stale closure issues.
- **Complete test coverage for core library**: `client.ts`, `events.ts`, `public-env.ts`, and `analytics-provider.tsx` all reach 100% coverage.
- **Onboarding completion test is thorough**: The `onboarding-wizard.test.tsx` test simulates the transition from incomplete → complete onboarding via `rerender`, correctly validating the "fire once" semantics of the completion event.
- **Promo engagement granularity**: Three distinct `action` values (`source_link_clicked`, `calculator_opened`, `calculator_calculated`) give the analytics funnel clear drop-off visibility without polluting the event namespace.
- **No `console.log` introduced**: All new files follow the Pino convention (no logging needed in client-only analytics code, which is appropriate).
