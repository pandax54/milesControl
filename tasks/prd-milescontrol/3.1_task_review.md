## Summary

### 🟢 Minor Issues

**1. Unreachable null-coalescing fallback on line 228**
- **File**: `src/lib/scrapers/passageiro-de-primeira.ts`, line 228
- **Description**: The expression `foundPrograms[1] ?? foundPrograms[0]` contains a dead fallback. At this point in the function, `foundPrograms.length` is guaranteed to be >= 2 (length 0 returns `{}` on line 187, length 1 returns on line 192). Therefore `foundPrograms[1]` is always defined and `?? foundPrograms[0]` can never be reached. This is harmless defensive coding and explains the uncoverable branch at line 228 in the coverage report.
- **Impact**: No behavioral impact. Coverage gap at line 228 is expected and cannot be covered.
- **Suggested fix**: No action needed. Optionally simplify to `destination: foundPrograms[1]` if 100% branch coverage is desired, but the defensive fallback is reasonable.

**2. Weak assertion in constructor default test**
- **File**: `src/lib/scrapers/passageiro-de-primeira.test.ts`, lines 410-414
- **Description**: The test `should use default config values` only asserts that `createScraperRun` has not been called, which verifies that construction doesn't trigger a run — but doesn't verify any default config values. This is a very minor gap since the integration tests through `run()` effectively exercise the defaults (correct URL, source name, etc.).
- **Impact**: Very low. The integration tests cover this indirectly.
- **Suggested fix**: No action needed. The test name could optionally be renamed to `should construct without triggering a run` to better match what it asserts.
