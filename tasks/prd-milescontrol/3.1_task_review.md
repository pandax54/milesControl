## ✅ Positive Highlights

- **Clean BaseScraper extension**: The scraper class is minimal (~40 lines), only overriding `extractPromotions`. All parsing logic lives in pure, individually exported helper functions — highly testable and reusable for future scrapers.
- **Well-crafted regex patterns**: The bonus, discount, transfer, and reverse patterns are purpose-built for Portuguese-language promotion titles, with appropriate use of non-capturing groups, Unicode character classes, and case-insensitive flags.
- **Substring deduplication in program extraction**: The `rawMatches.filter()` logic correctly handles the "Azul" vs "Azul Fidelidade" overlap. Well-documented with a comment and tested.
- **Typed mock factories**: `buildMockScraperRun` uses `ScraperRun` type from Prisma with `Partial<ScraperRun>` overrides, consistent with the project-wide convention established in tasks 2.6–3.0.
- **No `as any`, no `as never`, no `console.log`, no `process.env` access**: Full compliance with project rules.
- **Realistic HTML fixtures**: `buildArticleHtml` and `buildPageHtml` test helpers create HTML structures closely mirroring the actual blog's DOM, enabling realistic integration tests.
- **Complete integration tests through `BaseScraper.run()`**: The test suite exercises the full scraper lifecycle (robots.txt check → rate limiting → fetch → extract → persist) with 11 integration scenarios plus 43 unit tests for individual helpers.
- **Well-chosen constants**: `SOURCE_NAME`, `BASE_URL`, `SCRAPER_PATH`, `PROMO_CATEGORIES`, `KNOWN_PROGRAMS`, and all regex patterns are extracted to named module-level constants. No magic values.
- **Proper cleanup**: `vi.unstubAllGlobals()` in `afterEach` correctly cleans up `vi.stubGlobal('fetch', ...)` stubs.
- **Pragmatic scraping strategy**: Scraping `/` instead of `/category/promocoes` due to anti-bot protection on the category page, with category-based filtering to extract relevant articles.
