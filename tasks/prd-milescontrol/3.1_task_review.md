### Minor Issues

**1. Program extraction order depends on `KNOWN_PROGRAMS` array order, not title position**
- **File**: `src/lib/scrapers/passageiro-de-primeira.ts`, line 179-183 and 230
- **Description**: The `extractPrograms` function iterates `KNOWN_PROGRAMS` in its defined order and pushes matches into `rawMatches`. When the final fallback (line 230) assigns `foundPrograms[0]` as source and `foundPrograms[1]` as destination, it reflects the order programs appear in the constant array, not the order they appear in the title text. For example, `"Livelo e Smiles com bonus"` assigns `source: 'Smiles'` and `destination: 'Livelo'` because Smiles appears before Livelo in the `KNOWN_PROGRAMS` array. This is tested and intentional, but could produce surprising results if the array is reordered. A "why" comment at the fallback explaining this behavior would improve maintainability.
- **Suggested fix**: Add a comment at line 229-230:
```typescript
// Fallback order follows KNOWN_PROGRAMS array order, not title position.
// This is acceptable since transfer direction is already handled by
// explicit patterns above; this branch is for ambiguous titles.
return { source: foundPrograms[0], destination: foundPrograms[1] ?? foundPrograms[0] };
```

**2. Uncovered branch: `extractDiscountPercent` with non-finite or zero value**
- **File**: `src/lib/scrapers/passageiro-de-primeira.ts`, line 172
- **Description**: The `Number.isFinite(value) && value > 0` guard on line 172 has its `false` branch uncovered. While the equivalent guard in `extractBonusPercent` (line 163) is covered by the `'Programa com 0% de bonus'` test, `extractDiscountPercent` has no test for `0%` discount. The branch coverage report confirms line 172 is uncovered. This is a very minor gap since the code path is structurally identical to the bonus function.
- **Suggested fix**: Add a test:
```typescript
it('should return undefined for 0% discount', () => {
  expect(extractDiscountPercent('Pontos com 0% de desconto')).toBeUndefined();
});
```

**3. Missing test for `'compre e pontue'` category**
- **File**: `src/lib/scrapers/passageiro-de-primeira.test.ts`
- **Description**: The `PROMO_CATEGORIES` set includes six categories, but `isPromotionRelevant` tests only cover five of them. The `'compre e pontue'` category is never tested. While coverage tools may not flag this since the `Set.has()` call is already exercised, it is good practice to have at least one assertion per member of a domain-significant set to catch typos or drift.
- **Suggested fix**: Add a test case:
```typescript
it('should return true for compre e pontue category', () => {
  expect(isPromotionRelevant(['Compre e pontue'])).toBe(true);
});
```

**4. Uncovered branches in `extractPrograms` reverse pattern fallback (lines 224-230)**
- **File**: `src/lib/scrapers/passageiro-de-primeira.ts`, lines 210, 224-230
- **Description**: Coverage reports show lines 210, 224-230 as uncovered branches. Line 210 is the case where the transfer pattern matches but the extracted source/destination parts don't contain any of the `foundPrograms`. Lines 224-230 are the case where the reverse pattern matches but the source/destination can't be resolved. These are defensive fallthrough paths that are hard to trigger with realistic titles, but adding one edge-case test for each would improve confidence.
- **Suggested fix**: Add tests that exercise these fallthrough paths, e.g., a title like `"Transfira pontos do banco para outro programa"` with two programs that don't appear in the capture groups.

**5. Constructor could use object spread for conciseness**
- **File**: `src/lib/scrapers/passageiro-de-primeira.ts`, lines 52-63
- **Description**: The constructor manually maps each optional field from `configOverrides` to the `super()` call. This is safe but verbose. Since `BaseScraper` already applies `??` defaults for all optional fields, a simpler approach would be:
```typescript
constructor(configOverrides?: Partial<ScraperConfig>) {
  super({
    name: SOURCE_NAME,
    baseUrl: BASE_URL,
    scraperPath: SCRAPER_PATH,
    ...configOverrides,
  });
}
```
This achieves the same result with less code. The override-last spread ensures any provided values win, while `BaseScraper` applies defaults for `undefined` optionals.
- **Impact**: Style only; no behavioral difference.

## Recommendations

1. Add a "why" comment at the final fallback in `extractPrograms` (line 229-230) explaining that the ordering follows `KNOWN_PROGRAMS` array order, not title position. This prevents confusion if the array is later reordered.

2. Add a test for `extractDiscountPercent` with `0%` input to cover the remaining uncovered branch on line 172, achieving parity with the equivalent `extractBonusPercent` test.

3. Add a test for the `'compre e pontue'` category in `isPromotionRelevant` to ensure all six members of `PROMO_CATEGORIES` have at least one assertion.

4. Consider simplifying the constructor to use object spread (`{ name: SOURCE_NAME, baseUrl: BASE_URL, scraperPath: SCRAPER_PATH, ...configOverrides }`) for conciseness. This is a style preference and non-blocking.

5. Consider adding edge-case tests for the uncovered defensive fallthrough paths in `extractPrograms` (lines 210, 224-230) where pattern matches succeed but program name lookups fail. These are unlikely in practice but would bring branch coverage to near 100%.
