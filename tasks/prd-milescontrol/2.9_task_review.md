## Issues Found

### Major Issues

**1. Email address logged as `to` bypasses Pino redaction (resend.ts, lines 29, 41, 45)**

The logger redact configuration in `src/lib/logger.ts` redacts the `email` property, but the Resend integration logs the email address under the `to` property. This means user email addresses will appear in plain text in production logs.

```typescript
// Current: logs email as `to`, which is not redacted
logger.info({ to: params.to, subject: params.subject }, 'Email sent successfully');

// Fix option A: rename to `email` so redaction applies
logger.info({ email: params.to, subject: params.subject }, 'Email sent successfully');

// Fix option B: add `to` to the redact paths in logger.ts
redact: {
  paths: ['password', 'passwordHash', 'token', 'secret', 'creditCard', 'email', 'to'],
  censor: '[REDACTED]',
},
```

Option A is simpler and avoids broadening the redact config. However, if `to` is used elsewhere as a non-sensitive field, Option B with a more specific path like `*.to` could be considered.

**2. No test file for the cron route handler (src/app/api/cron/balance-digest/route.ts)**

The cron endpoint has no corresponding test file. While it is a thin layer, it contains non-trivial logic: the `verifyCronSecret` function with its development fallback (allowing all requests when `CRON_SECRET` is unset). This should be covered by tests verifying:
- Request without authorization header returns 401 when `CRON_SECRET` is set
- Request with correct bearer token succeeds
- Request succeeds when `CRON_SECRET` is not configured (dev mode)
- Error handling returns 500 on service failure

### Minor Issues

**3. `verifyCronSecret` allows unauthenticated access when CRON_SECRET is unset (route.ts, line 29)**

When `CRON_SECRET` is falsy (empty string or undefined), the function returns `true`, allowing unauthenticated access. While the comment says "Allow in development," there is no environment check -- this behavior would apply in production too if someone forgets to set the variable. Consider gating on `IS_DEVELOPMENT`:

```typescript
import { IS_DEVELOPMENT } from '@/lib/env';

function verifyCronSecret(request: NextRequest): boolean {
  if (!env.CRON_SECRET) {
    if (IS_DEVELOPMENT) {
      return true;
    }
    logger.error('CRON_SECRET is not configured in non-development environment');
    return false;
  }
  const authHeader = request.headers.get('authorization');
  return authHeader === `Bearer ${env.CRON_SECRET}`;
}
```

**4. `sendAllDigests` processes users sequentially (balance-digest.service.ts, line 223)**

The `sendAllDigests` function iterates users with `for...of` and `await`s each `sendDigestForUser` call. For a small user base this is fine, but as the system grows this will be slow. Consider batching with `Promise.allSettled` or using a concurrency-limited approach (e.g., processing 5 users at a time).

This is not blocking since the project is early stage.

**5. `takeBalanceSnapshots` snapshots all users' enrollments, not scoped per-user (balance-digest.service.ts, line 24)**

The function fetches all enrollments across all users with no filter. This works correctly but could grow into a performance concern at scale. Consider batching by user or using a cursor-based approach for large datasets. Non-blocking for current scope.

**6. `programName` denormalization in BalanceSnapshot model (schema.prisma, line 493)**

The `programName` field on `BalanceSnapshot` is denormalized from the `Program` table. If a program is renamed, historical snapshots will show the old name while current data shows the new name. This is a deliberate trade-off for query simplicity and is acceptable, but worth documenting as a known behavior. A `programId` relation would allow joining to get the current name when needed -- though the model does have `programId`, it lacks a `@relation` directive to `Program`.

**7. HTML template uses inline styles without extraction (balance-digest.service.ts, lines 91-130)**

The HTML email template uses inline styles throughout, which is standard practice for email HTML (email clients strip `<style>` tags). However, the color constants (`#16a34a`, `#dc2626`, `#666`, `#1a1a1a`, `#f8f9fa`, `#dee2e6`) are magic values. Consider extracting them as named constants:

```typescript
const DIGEST_COLORS = {
  POSITIVE: '#16a34a',
  NEGATIVE: '#dc2626',
  NEUTRAL: '#666',
} as const;
```

## Recommendations

1. **Fix email redaction** (Major): Either log the recipient email under the `email` key so Pino redaction applies, or add `to` to the redact paths in `logger.ts`. This prevents PII from appearing in production logs.

2. **Add cron route tests** (Major): Create `src/app/api/cron/balance-digest/route.test.ts` covering the authorization check and error handling paths.

3. **Guard cron access in production** (Minor): Add an `IS_DEVELOPMENT` check to the `verifyCronSecret` fallback so a missing `CRON_SECRET` does not silently allow unauthenticated access in production.

4. **Extract color constants** (Minor): Replace magic color hex values in `resolveChangeColor` and the HTML template with named constants.

5. **Consider adding a `Program` relation to `BalanceSnapshot`** (Minor): While `programName` denormalization is a valid choice, adding `program Program @relation(...)` would allow future queries to join against the canonical program data.

## Verdict

The implementation is solid and well-crafted. The service architecture is clean with proper separation between data access, business logic, and email rendering. Test coverage is thorough at 100% for both the service and integration files. The two major observations -- email PII in logs and missing cron route tests -- are non-blocking but should be addressed before production deployment. Approved with observations.
