### рџџў Minor Issues

**1. `as never` casts in test mocks** вЂ” `balance-digest.service.test.ts` lines 232, 267, 289, 335, etc.

Multiple `as never` casts are used for mock return values. Previous tasks (2.6, 2.7) established a pattern of replacing these with typed mock factories (e.g., `buildMockTransfer()`, `buildEnrollment()`). This test file should follow the same pattern for consistency.

**Suggested fix**: Create typed helper functions like `buildMockEnrollmentWithProgram()` and `buildMockSnapshot()` that return properly typed objects, eliminating the need for `as never` casts.

**2. No test for `fetchLatestSnapshots` deduplication with multiple snapshots** вЂ” `balance-digest.service.ts` lines 191вЂ“213

The `fetchLatestSnapshots` function deduplicates by keeping the earliest snapshot per program from a descending-ordered query. While tested indirectly via `sendDigestForUser`, a dedicated test with multiple snapshots per program would improve confidence in the deduplication behavior. Coverage shows line 213 uncovered.

**3. Missing cron route test** вЂ” `src/app/api/cron/balance-digest/route.ts`

The cron endpoint has no corresponding test file. While it is a thin layer, it contains non-trivial authorization logic (`verifyCronSecret` with dev fallback). A test would verify:
- Request without authorization header returns 401 when `CRON_SECRET` is set
- Request with correct bearer token succeeds
- Development fallback behavior
- Error handling returns 500 on service failure

**4. `programName` denormalization without `Program` relation** вЂ” `schema.prisma` line 493

The `BalanceSnapshot` model has `programId` and `programName` but no `@relation` to `Program`. If a program is renamed, historical snapshots will show the old name. This is an acceptable trade-off for query simplicity, but adding a `Program` relation would allow future queries to join against canonical data.
