## Summary

Task 3.0 implements the foundational scraper infrastructure for Phase 3 (Promotions Engine). The implementation delivers: shared types and constants (`types.ts`), robots.txt fetching/parsing/caching (`robots-txt.ts`), per-source rate limiting (`rate-limiter.ts`), an abstract `BaseScraper` class with retry logic and ScraperRun persistence (`base-scraper.ts`), and a CRUD/query service for ScraperRun records (`scraper-run.service.ts`).

Overall quality is high. The code is well-structured, uses proper TypeScript idioms (`readonly`, union types, interfaces), follows the project's Pino logging conventions, and has thorough test coverage (55 new tests across 4 test files, all passing). The architecture cleanly separates concerns and provides a solid foundation for concrete scraper implementations in tasks 3.1-3.4.

All previous review feedback has been addressed: sleep duplication eliminated, User-Agent constant unified, persistence error handling wrapped in try/catch, promotions array made readonly, network errors cached in robots.txt, /logs added to .gitignore, and `as ScraperRun` casts replaced with typed factory.

### 🟢 Minor Issues

1. **`as unknown as Response` casts in robots-txt.test.ts** (lines 282, 301, 315, 327, 339, 356) — Six `as unknown as Response` double-casts are used for fetch mock responses. While this is a standard pattern for fetch mocking, the project convention established in tasks 2.6-2.11 is to use typed mock factory functions. Consider extracting a `buildMockFetchResponse(overrides)` helper for consistency, similar to how `base-scraper.test.ts` already has `mockSuccessfulFetch()`.

   **Suggested fix:**
   ```typescript
   function buildMockResponse(overrides: { ok?: boolean; status?: number; text?: string } = {}): Response {
     return {
       ok: overrides.ok ?? true,
       status: overrides.status ?? 200,
       text: vi.fn().mockResolvedValue(overrides.text ?? ''),
     } as unknown as Response;
   }
   ```

## ✅ Positive Highlights

1. **Excellent use of `readonly`**: All interface properties in `ScrapedPromotion`, `ScraperConfig`, `RobotsRule`, `CompleteScraperRunInput`, `ScraperResult`, and all class fields in `BaseScraper` consistently use `readonly`. The `promotions` array in `ScraperResult` is `readonly ScrapedPromotion[]` for deep immutability.

2. **Proper Pino logging patterns**: All logging follows the data-first convention (`logger.info({ scraperRunId, sourceName }, 'message')`), errors use the `err` property, and child loggers are used for scraper-scoped context in `BaseScraper`. No `console.log` anywhere.

3. **Well-designed abstract class**: The `BaseScraper` provides a clean template method pattern — `run()` orchestrates the workflow (robots.txt check -> rate limit -> fetch with retry -> extract) while subclasses only need to implement `extractPromotions()`. This is a solid design for the planned concrete scrapers.

4. **Robust error handling**: The `run()` catch block wraps `completeScraperRun` in a nested try/catch so that a database error during persistence never masks the original scraping error. The `ScraperRun` record status is correctly set for all outcomes (SUCCESS, FAILED, BLOCKED).

5. **Comprehensive test coverage**: 55 new tests across 4 test files. The robots.txt parser has 10 tests covering edge cases (empty content, invalid crawl-delay, directives before user-agent, lines without colon). Rate limiter tests use fake timers correctly. The `BaseScraper` tests cover the full lifecycle including retry success, blocked paths, and persistence failure recovery.

6. **Proper exponential backoff**: The retry mechanism uses `retryDelayMs * Math.pow(2, attempt - 1)` providing genuine exponential backoff (50ms -> 100ms -> 200ms, etc.).

7. **Shared `sleep` utility**: The `sleep` function is defined once in `rate-limiter.ts` and imported by `base-scraper.ts`, avoiding duplication.

8. **Consistent User-Agent**: `fetchRobotsTxt` uses the shared `DEFAULT_USER_AGENT` constant from `types.ts`, matching the value used by `BaseScraper`.

9. **Network error caching**: Both HTTP errors and network errors in `fetchRobotsTxt` cache the empty result, preventing repeated timeouts for the same domain within the TTL window.

10. **Smart use of `select` in queries**: `countConsecutiveFailures` uses `select: { status: true }` to fetch only the needed field. The composite index `@@index([sourceName, startedAt])` on `ScraperRun` ensures efficient queries.

11. **No `any` types**: Zero uses of `any` across all new source and test files.

12. **Typed mock factories**: Both `base-scraper.test.ts` and `scraper-run.service.test.ts` use `buildMockScraperRun()` typed factories returning proper `ScraperRun` types, consistent with the established project pattern.

13. **Named constants throughout**: All magic numbers are extracted to well-named constants (`DEFAULT_DELAY_MS`, `DEFAULT_MAX_RETRIES`, `CONSECUTIVE_FAILURE_THRESHOLD`, `DEFAULT_FETCH_TIMEOUT_MS`, `DEFAULT_ROBOTS_TXT_TIMEOUT_MS`, etc.).
