#### 1. Duplicated `sleep` helper function

**Files**: `src/lib/scrapers/rate-limiter.ts` (line 24) and `src/lib/scrapers/base-scraper.ts` (line 165)
**Description**: The `sleep` function is identically defined in two files within the same directory. Per the code standards rule ("extract shared logic into a third module"), this should be a shared utility.
**Suggested fix**: Extract `sleep` into a shared utility, either in `src/lib/utils/` or as an exported function in `types.ts` (since it is scraper-specific). Both files would then import from that single location.

```typescript
// src/lib/scrapers/types.ts (or src/lib/utils/async.ts)
export function sleep(ms: number): Promise<void> {
  return new Promise(resolve => setTimeout(resolve, ms));
}
```

#### 2. Hardcoded User-Agent in `fetchRobotsTxt`

**File**: `src/lib/scrapers/robots-txt.ts` (line 41)
**Description**: The User-Agent header in the `fetch` call uses the hardcoded string `'MilesControl/1.0'`, while `types.ts` defines `DEFAULT_USER_AGENT = 'MilesControl/1.0 (+https://milescontrol.com)'`. These values diverge. The `fetchRobotsTxt` function does not accept a user-agent parameter, so it cannot use the configured value from `BaseScraper`.
**Suggested fix**: Either add a `userAgent` parameter to `fetchRobotsTxt` and thread it through from the caller, or import and use `DEFAULT_USER_AGENT` for consistency.

```typescript
export async function fetchRobotsTxt(
  baseUrl: string,
  cacheTtlMs = DEFAULT_ROBOTS_TXT_CACHE_TTL_MS,
  userAgent = DEFAULT_USER_AGENT,
): Promise<readonly RobotsRule[]> {
  // ...
  const response = await fetch(url, {
    headers: { 'User-Agent': userAgent },
    signal: AbortSignal.timeout(DEFAULT_ROBOTS_TXT_TIMEOUT_MS),
  });
```

#### 3. `completeScraperRun` failure in error handler is unhandled

**File**: `src/lib/scrapers/base-scraper.ts` (lines 87-101)
**Description**: In the `catch` block of `run()`, if `completeScraperRun` throws (e.g., database connection lost), that error will propagate unhandled and replace the original scraping error. This could cause a scraper failure to surface as a database error instead, making debugging harder. The `ScraperRun` record would also be left in `RUNNING` status permanently.
**Suggested fix**: Wrap the persistence call in a try/catch so that the original error context is preserved.

```typescript
} catch (error) {
  const errorMessage = error instanceof Error ? error.message : 'Unknown error';
  this.log.error({ err: error }, 'Scraping failed');

  try {
    await completeScraperRun({
      id: scraperRun.id,
      status: SCRAPER_RUN_STATUS.FAILED,
      itemsFound: 0,
      newPromos: 0,
      startedAt,
      errorMessage,
    });
  } catch (persistError) {
    this.log.error({ err: persistError, scraperRunId: scraperRun.id }, 'Failed to persist scraper run failure');
  }

  return { promotions: [], scraperRunId: scraperRun.id };
}
```

#### 4. `ScraperResult.promotions` should use `readonly` array

**File**: `src/lib/scrapers/base-scraper.ts` (line 21)
**Description**: The `ScraperResult` interface marks `promotions` as `readonly` at the property level, but the array type is `ScrapedPromotion[]` (mutable). Given that `ScrapedPromotion` already uses `readonly` on all its fields, using `readonly ScrapedPromotion[]` would be more consistent with the project's immutability preferences.
**Suggested fix**:

```typescript
export interface ScraperResult {
  readonly promotions: readonly ScrapedPromotion[];
  readonly scraperRunId: string;
}
```

#### 5. Network error results are not cached in `fetchRobotsTxt`

**File**: `src/lib/scrapers/robots-txt.ts` (lines 58-61)
**Description**: When a network error occurs (e.g., DNS failure, timeout), the `catch` block returns an empty array but does not cache the result. This means every subsequent call for the same `baseUrl` will retry the fetch, potentially causing repeated timeouts. Compare with the HTTP error path (line 48) which correctly caches empty rules. This is a minor inconsistency -- depending on the desired behavior, caching network errors may or may not be preferred (failing to cache could be seen as a feature for transient network issues, but for robots.txt it is unlikely to resolve within the same run).
**Suggested fix**: Consider caching the empty result on network error too, or add a comment explaining why network errors are intentionally not cached.

#### 6. Missing explicit return types on service functions

**File**: `src/lib/services/scraper-run.service.ts` (lines 8, 32, 58, 66)
**Description**: The functions `createScraperRun`, `completeScraperRun`, `listScraperRuns`, and `getLatestRunBySource` all rely on inferred return types from Prisma calls. While this works due to TypeScript inference, adding explicit return type annotations (e.g., `Promise<ScraperRun>`) would make the public API surface clearer and protect against accidental signature changes. Note: this is consistent with the existing codebase pattern where most services also omit return types, so this is a very minor observation rather than a violation.

7. **Named constants throughout**: All magic numbers are extracted to well-named constants (`DEFAULT_DELAY_MS`, `DEFAULT_MAX_RETRIES`, `CONSECUTIVE_FAILURE_THRESHOLD`, etc.).

8. **No `any` types**: Zero uses of `any` across all new files. `unknown` is not needed either because the code properly handles error types with `instanceof Error` checks.

9. **Clean module structure**: Good separation of concerns -- types, robots.txt handling, rate limiting, base class, and persistence service are each in their own file with clear responsibilities.

## Recommendations

1. **Extract the duplicated `sleep` helper** to a single shared location within `src/lib/scrapers/` or `src/lib/utils/`. This is the most straightforward improvement.

2. **Align the User-Agent string** in `fetchRobotsTxt` with `DEFAULT_USER_AGENT` from `types.ts` to avoid subtle inconsistency. Consider parameterizing `fetchRobotsTxt` to accept a user-agent.

3. **Guard against persistence failure** in the `run()` catch block. A try/catch around `completeScraperRun` in the error path would prevent a database error from masking the original scraping error.

4. **Add `logs/` to `.gitignore`**. The `logs/phase-3/` directory appears in untracked files and likely should not be committed to the repository.

5. **Consider caching network errors** in `fetchRobotsTxt` for consistency with the HTTP error path, or add an explanatory comment about the intentional difference.
