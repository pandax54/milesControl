### Minor Issues

**1. Authentication error is swallowed as a generic error in Server Actions**
- **File**: `src/actions/programs.ts`, lines 42-55
- **Description**: When `requireUserId()` throws `"Not authenticated"`, it falls into the generic catch block and returns `"Failed to enroll. Please try again."` (and similar for edit/remove). The test on line 136-143 confirms this behavior. While not a security issue (the operation is blocked), the user receives a misleading error message instead of being told they are not authenticated.
- **Suggestion**: Catch the authentication error explicitly and return a more specific message, or redirect to login.

```typescript
if (error instanceof Error && error.message === 'Not authenticated') {
  return { success: false, error: 'You must be logged in to perform this action' };
}
```

**2. Balance parsing with `parseInt(...) || 0` silently converts NaN to 0**
- **File**: `src/components/dashboard/enrollment-form-dialog.tsx`, line 61; `src/components/dashboard/edit-enrollment-dialog.tsx`, line 53
- **Description**: `parseInt(currentBalance, 10) || 0` means if a user types a non-numeric string (e.g., "abc"), it silently converts to 0. This is not dangerous since the Zod schema validates on the server, but it may confuse users who expect client-side feedback.
- **Suggestion**: Consider adding basic client-side validation or using `Number()` with a validation check before submission.

**3. `createEnrollment` performs two sequential queries that could be a single query**
- **File**: `src/lib/services/program-enrollment.service.ts`, lines 39-58
- **Description**: The function first checks for existing enrollment, then checks if the program exists, using two separate queries. Under normal conditions this is fine, but in a race condition scenario, two concurrent requests could both pass the duplicate check. Prisma's unique constraint (`@@unique([userId, programId])`) will catch this at the database level, but the error would surface as an unhandled Prisma error rather than the user-friendly `EnrollmentAlreadyExistsError`.
- **Suggestion**: Wrap the create in a try/catch that catches Prisma's unique constraint violation error (P2002) and converts it to `EnrollmentAlreadyExistsError`:

```typescript
try {
  const enrollment = await prisma.programEnrollment.create({ ... });
  return enrollment;
} catch (error) {
  if (error instanceof Prisma.PrismaClientKnownRequestError && error.code === 'P2002') {
    throw new EnrollmentAlreadyExistsError(input.programId);
  }
  throw error;
}
```

**4. Staleness thresholds use magic numbers**
- **File**: `src/components/dashboard/enrollment-card.tsx`, lines 37-40
- **Description**: The staleness thresholds (7 and 30 days) are inline magic numbers. Per project standards (code-standards.md), these should be named constants.
- **Suggestion**: Extract to named constants:

```typescript
const STALENESS_FRESH_DAYS = 7;
const STALENESS_WARNING_DAYS = 30;
```

## Recommendations

1. **Extract staleness threshold constants** -- Move the 7-day and 30-day values to named constants in the enrollment card or a shared constants file, as they will also be needed by Task 2.1 (quick-update UX).

2. **Handle Prisma unique constraint race condition** -- Add a catch for Prisma error code P2002 in `createEnrollment` to gracefully handle concurrent enrollment attempts, rather than relying solely on the pre-check query.

3. **Differentiate authentication errors in Server Actions** -- Return a specific "not authenticated" error message when `requireUserId()` fails, rather than the generic "Please try again" message. This improves the user experience and debugging.

4. **Consider client-side validation** -- The form components could benefit from lightweight client-side validation (e.g., preventing non-numeric balance input) to provide faster feedback, even though server-side Zod validation is the authoritative check.
