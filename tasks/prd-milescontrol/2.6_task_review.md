### Minor Issues

1. **Edit dialog does not allow changing source/destination programs** (`src/components/dashboard/edit-transfer-dialog.tsx`)

   The edit dialog only exposes fields for points, bonus, miles, cost, date, and notes. It does not allow the user to change the source or destination program. While the update schema and service support these fields, the UI does not. If a user logs a transfer with the wrong source or destination, they would need to delete it and re-create it. This is a minor UX gap -- consider adding program selects to the edit dialog, which would require passing `programs` as a prop.

2. **`updateField` uses untyped `string` key** (`src/components/dashboard/edit-transfer-dialog.tsx`, line 49)

   ```typescript
   function updateField(field: string, value: string) {
   ```

   The `field` parameter should be typed as `keyof typeof formState` (or an equivalent union type) to get compile-time safety against typos.

3. **`costPerMilheiro` is always recalculated on update, even when unchanged** (`src/lib/services/transfer.service.ts`, lines 61-64)

   When updating a transfer where neither `totalCost` nor `milesReceived` changed, the service still recalculates `costPerMilheiro` and includes it in the update payload. This is functionally correct but slightly wasteful. Minor -- no action required.

4. **`TransferData` interface exported from a page component** (`src/app/(dashboard)/transfers/page.tsx`, line 20)

   The `TransferData` interface is exported from the page file and imported by `edit-transfer-dialog.tsx`. Shared types are better placed in a dedicated types file or in the validators module to avoid coupling a component to a page file. This follows the existing project pattern seen in other areas, so it is not blocking.

5. **Rating thresholds are defined locally in the page** (`src/app/(dashboard)/transfers/page.tsx`, lines 34-36)

   The rating thresholds (Excellent < R$12, Good R$12-16, Acceptable R$16-20, Avoid > R$20) match the domain concepts from CLAUDE.md. These same thresholds will likely be needed in the dashboard component (`dashboard-transfers.tsx`) and the future Miles Value Advisor (task 3.8). Consider extracting them to a shared constants file (e.g., `src/lib/constants/rating.ts`) to avoid duplication later.

6. **`DialogTrigger` uses `render` prop pattern** (`src/components/dashboard/transfer-form-dialog.tsx`, line 111)

   ```tsx
   <DialogTrigger render={<Button />}>
   ```

   This uses the `render` prop of `DialogTrigger`, which is a valid pattern in some shadcn/ui versions but may be fragile. Consistent with other dialog components in the project, so no action needed -- just noting for awareness.

7. **Missing `transferDate` validation for format** (`src/lib/validators/transfer.schema.ts`, line 12)

   The `transferDate` field is typed as `z.string().optional()` but has no format validation. An invalid date string like `"not-a-date"` would pass Zod validation and then produce an `Invalid Date` via `new Date(input.transferDate)` in the service. Consider adding `.regex(/^\d{4}-\d{2}-\d{2}$/)` or a `.refine()` to validate the ISO date format.

## Recommendations

1. **Extract `TransferData` interface** to a shared types file (e.g., `src/lib/types/transfer.ts`) to decouple the edit dialog from the page component.
2. **Extract rating constants** (`COST_THRESHOLD_EXCELLENT`, etc.) to a shared module for reuse in the dashboard and future Miles Value Advisor feature.
3. **Add date format validation** to `transferDate` in the Zod schema to prevent invalid date strings from reaching the service layer.
4. **Add source/destination program editing** to the edit transfer dialog for a complete editing experience.
5. **Type the `updateField` parameter** with a union type matching the form state keys instead of plain `string`.
