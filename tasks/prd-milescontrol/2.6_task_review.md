### 🟢 Minor Issues

1. **`as never` casts in test mocks** — `transfer.service.test.ts` has ~15 `as never` casts and `transfers.test.ts` has ~8. The service test uses a properly typed `buildMockTransfer()` factory with `MockTransferLog` interface (good), but mock return values still use `as never` (e.g., `mockFindMany.mockResolvedValue(mockTransfers as never)`). This pattern was flagged and fixed in prior reviews (2.4, 2.5).

   **Suggested fix**: Cast factory results to the Prisma `TransferLog` type when passing to mocked functions. In the actions test, type `MOCK_SESSION` to match the NextAuth `Session` type.

2. **Duplicated form validation logic** — `transfer-form-dialog.tsx:61-86` and `edit-transfer-dialog.tsx:55-80` contain nearly identical client-side parsing/validation for points, miles, bonus, and cost fields. This mirrors the pattern previously fixed in task 2.4 where a shared `CreditCardFormFields` component was extracted.

   **Suggested fix**: Extract a shared `parseTransferFormValues(formState)` utility function, or a shared `TransferFormFields` component, to deduplicate.

3. **Schema coverage gap on `updateTransferSchema` date refine** — `transfer.schema.ts` is at 80% coverage; line 29 (the `transferDate` refine in `updateTransferSchema`) is uncovered. While the same refine is tested in `createTransferSchema`, adding one test for invalid date on the update schema would close the gap.

   **Suggested fix**: Add to `updateTransferSchema` tests:
   ```typescript
   it('should reject invalid transferDate', () => {
     const result = updateTransferSchema.safeParse({
       transferId: 'transfer-123',
       transferDate: 'not-a-date',
     });
     expect(result.success).toBe(false);
   });
   ```

4. **Rating thresholds duplicated** — `page.tsx:21-23` defines `COST_THRESHOLD_EXCELLENT/GOOD/ACCEPTABLE` which also appear in `dashboard-transfers.tsx`. Already deferred to task 3.6 where shared `cost-rating.ts` utilities will be created. No action needed now.

## ✅ Positive Highlights

1. **Clean layered architecture** — Validators, service, server actions, and UI are properly separated with clear responsibilities at each layer.
2. **Cost-per-milheiro auto-calculation** — `calculateCostPerMilheiro()` is a pure function, well-tested with edge cases (zero, negative miles), and automatically applied on create and recalculated on update.
3. **Ownership checks** — Both `updateTransfer` and `deleteTransfer` verify the transfer belongs to the requesting user via `findFirst({ where: { id, userId } })` before mutating.
4. **Custom error class** — `TransferNotFoundError` enables clean error discrimination in server actions without exposing internals.
5. **Shared `TransferData` interface** — Defined in the validator module (`transfer.schema.ts`) and reused by the page and both dialog components, avoiding type duplication.
6. **Typed `FormField` in edit dialog** — Line 49 derives `FormField` from the return type of `buildInitialState`, ensuring `updateField` only accepts valid field keys.
7. **Date validation in Zod schemas** — Both create and update schemas include `.refine()` to reject invalid date strings before they reach the service layer.
8. **Edit dialog resets on open** — `handleOpen` rebuilds form state from the transfer prop each time, preventing stale data.
9. **Proper Pino logging** — All service operations log structured context (`userId`, `transferId`) with data-first format. No `console.log` anywhere.
10. **Decimal→number conversion at page boundary** — `page.tsx:61-62` converts Prisma Decimal to number before passing to components, keeping UI layer clean of ORM types.
