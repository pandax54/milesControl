## Positive Highlights

- **UTC-consistent date handling end-to-end.** `buildWhereClause` constructs month boundaries with `Date.UTC(year, month - 1, 1)`. `getUtcMidnightToday()` uses `setUTCHours(0, 0, 0, 0)`. `groupEventsByMonth` reads year/month via `getUTCFullYear()` / `getUTCMonth()`. `formatDateRange` passes `timeZone: 'UTC'` to `Intl.DateTimeFormat` and uses `getUTCDate()` / `getUTCMonth()` for the same-month branch. All operations are self-consistent and will not produce off-by-one-day errors for any user regardless of their local timezone.

- **Single source of truth for UTC midnight.** Exporting `getUtcMidnightToday()` from the service and having both `miles-calendar-event-card.tsx` and `miles-calendar-view.tsx` import it directly eliminates the risk of drift between the service filter and the UI "upcoming" classification. The page, card, and view will always agree on what "today" means.

- **`buildWhereClause` precedence is correct.** When both `year+month` and `upcomingOnly` are provided, the year/month filter wins (checked first). The function does not combine both conditions, which is the right behavior since `listEventsByMonth` passes only year/month and never `upcomingOnly`.

- **Exhaustive `TYPE_BADGE_VARIANTS` map.** `Record<PromoType, 'default' | 'secondary' | 'outline'>` is exhaustive — if a new `PromoType` enum value is added to the Prisma schema and regenerated, TypeScript will surface a compile error at this map immediately rather than silently falling back to `undefined`.

- **Null guard on `event.expectedType`.** The card checks `event.expectedType &&` before indexing into `TYPE_BADGE_VARIANTS`, correctly handling the nullable Prisma field without a non-null assertion.

- **Functional grouping in `groupEventsByMonth`.** The `events.reduce<Map<string, EventsByMonth>>(...)` pattern is idiomatic, avoids `for` loops and mutation outside the accumulator, and is fully type-safe. The result is a `Map` iterated in insertion order, which preserves the ascending `startDate` order the service query guarantees.

- **Correct Pino usage.** The single log call follows data-first, message-last convention: `logger.debug({ count: events.length, filters }, 'Fetched calendar events')`. Using `debug` level for a query result is appropriate — it will not produce noise in production.

- **100% service coverage.** Every exported function is exercised: `listCalendarEvents` (default call, limit, `upcomingOnly` with fake timers, year+month filter, no-`take` path), `listUpcomingCalendarEvents` (default limit, custom limit), `listEventsByMonth`, `groupEventsByMonth` (grouping, label formatting, empty array, event order within month), and `PROMO_TYPE_LABELS`. The test for `upcomingOnly` correctly uses `vi.useFakeTimers()` / `vi.useRealTimers()` cleanup.

- **Correct Vitest patterns.** The test file uses `vi.mocked()` for typed mock access, `vi.clearAllMocks()` in `beforeEach`, and `vi.useFakeTimers()` / `vi.useRealTimers()` for the time-dependent test. All three align with the project's testing rules.

- **`past` groups displayed in reverse chronological order.** `pastGroups.reverse()` in `miles-calendar-view.tsx` means the most recently elapsed month appears closest to the `Upcoming` section. This is the natural reading order for a calendar page and avoids requiring the user to scroll to the bottom to find events that expired last month.

- **PRD completeness.** Both specified features are present — a calendar view of expected promotional periods (upcoming section with month grouping) and historical notes displayed in a distinct muted info block with an `Info` icon.

- **No magic numbers.** The only numeric literal in the page component, `100`, is extracted as `MAX_CALENDAR_PAGE_EVENTS`. The service's default limit `50` is extracted as `DEFAULT_UPCOMING_LIMIT`. No raw integer literals appear anywhere in the business logic.

- **Clean `const`-only code.** Every variable in the service uses `const`. No `let` or `var` appears in the production files.

- **Import order.** All files follow the external → internal (`@/`) → relative ordering required by the code standards.
