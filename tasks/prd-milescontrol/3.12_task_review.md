### Minor Issues

**1. Timezone inconsistency between `buildWhereClause` and `groupEventsByMonth`**
File: `src/lib/services/miles-calendar.service.ts`, lines 72-73 vs 105-106

`buildWhereClause` constructs date boundaries using the local timezone constructor:
```ts
const monthStart = new Date(year, month - 1, 1); // LOCAL time midnight
const monthEnd = new Date(year, month, 1);        // LOCAL time midnight
```

`groupEventsByMonth` reads dates using UTC methods:
```ts
const year = date.getUTCFullYear();
const month = date.getUTCMonth() + 1;
```

In a UTC environment (Vercel production) these are equivalent and there is no bug. However, in a local development environment on a UTC-3 machine (the Brazilian target market), `new Date(2026, 10, 1)` creates `2026-11-01T03:00:00Z`, which would exclude events stored as `2026-11-01T00:00:00Z` when filtering. The `isEventUpcoming` in `miles-calendar-event-card.tsx` (line 46) and `splitEventsByTimeline` in `miles-calendar-view.tsx` (line 12) both use `setHours(0, 0, 0, 0)` on a local `Date` — the same inconsistency at the component level.

**Suggestion**: use UTC boundary construction consistently:
```ts
const monthStart = new Date(Date.UTC(year, month - 1, 1));
const monthEnd = new Date(Date.UTC(year, month, 1));
```
And for the "today" boundary:
```ts
const today = new Date();
today.setUTCHours(0, 0, 0, 0);
```

**2. Import order violation in `miles-calendar-view.tsx`**
File: `src/components/promotions/miles-calendar-view.tsx`, lines 1-3

The relative import appears before the internal `@/` imports:
```ts
import { MilesCalendarEventCard } from './miles-calendar-event-card'; // relative
import type { MilesCalendarEvent } from '@/generated/prisma/client';  // @/ internal
import { groupEventsByMonth } from '@/lib/services/miles-calendar.service'; // @/ internal
```

Per code standards, order should be: external → internal (`@/`) → relative. That said, this same pattern is used throughout the existing codebase (e.g., `promotion-card.tsx`, `promotion-feed.tsx`) so this is a pre-existing project-wide inconsistency rather than a new violation introduced by this task.

**3. `for...of` loops where array methods could be used**
Files: `src/lib/services/miles-calendar.service.ts` line 103; `src/components/promotions/miles-calendar-view.tsx` line 17

Code standards prefer `find`, `filter`, `map`, `reduce` over `for`/`while`. The `for...of` loop in `groupEventsByMonth` is building a `Map`, which is a legitimate case where `reduce` would be more verbose and less readable. The loop in `splitEventsByTimeline` partitions an array, which is also a common `reduce` pattern. These are low-priority since the `for...of` pattern is widespread across services in this project (e.g., `balance-digest.service.ts`, `promotion.service.ts`).

**4. Non-null assertion in `groupEventsByMonth`**
File: `src/lib/services/miles-calendar.service.ts`, line 118

```ts
groups.get(key)!.events.push(event);
```

The assertion is logically safe because the key is guaranteed to exist after the `set` call on lines 109-117, but non-null assertions suppress TypeScript's safety checks. A safer alternative that avoids the `!` is:
```ts
const group = groups.get(key);
if (group) {
  group.events.push(event);
}
```

## Recommendations

1. **Use UTC-consistent date boundaries** in `buildWhereClause` and the "today" comparisons in both `miles-calendar-event-card.tsx` and `miles-calendar-view.tsx`. The production environment is UTC so this has no user-facing impact today, but it will cause silent off-by-one bugs for any developer running on a non-UTC machine and if the service is ever called from a non-Vercel environment.

2. **Mark task 3.12 as complete** in `tasks/prd-milescontrol/tasks.md` by changing `- [ ] 3.12` to `- [x] 3.12`.

3. **Eliminate the non-null assertion** on line 118 of `miles-calendar.service.ts`. The `Map` lookup can be expressed without `!` by using a `const group = groups.get(key); if (group) { group.events.push(event); }` pattern that TypeScript can verify.

4. **Import order in `miles-calendar-view.tsx`** should be corrected to place the relative import after the `@/` imports. Although the same violation exists throughout the codebase, the new code should model the standard rather than replicate the pattern.
