### Minor Issues

**1. Import order in `miles-calendar-view.tsx`**
File: `src/components/promotions/miles-calendar-view.tsx`, lines 1-3

The relative import appears last (line 3), which is the correct order (external > internal `@/` > relative). However, the two `@/` imports on lines 1-2 themselves are fine. On closer inspection the order is: `@/generated` (line 1), `@/lib` (line 2), `./miles-calendar-event-card` (line 3) — this is actually correct per the standard. No issue.

**1. Page fetches 100 events with a magic-number limit**
File: `src/app/(dashboard)/promotions/calendar/page.tsx`, line 5

```ts
const events = await listCalendarEvents({ limit: 100 });
```

The value `100` is a magic number used directly in the page component. Per code standards, magic numbers must be extracted to a named constant. A named constant like `MAX_CALENDAR_PAGE_EVENTS = 100` at the module level would express intent and make future adjustments discoverable.

**2. `isEventUpcoming` duplicated between service component and view component**
Files: `src/components/promotions/miles-calendar-event-card.tsx` line 44 and `src/components/promotions/miles-calendar-view.tsx` line 11

Both `isEventUpcoming` (in the card) and `splitEventsByTimeline` (in the view) independently compute a UTC midnight "today" boundary:
```ts
const today = new Date();
today.setUTCHours(0, 0, 0, 0);
```

The duplication is minor since both files are small and co-located, but extracting this single-line boundary computation into a shared utility (or into the service) would eliminate the risk of the two going out of sync in the future.
