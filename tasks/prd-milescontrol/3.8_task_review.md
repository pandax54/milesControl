### 🟢 Minor Issues

**1. `cost-calculator.service.ts` is 407 lines (exceeds 300-line guideline)**

The file was close to the limit after tasks 3.6/3.7, and the ~107 lines added for the redemption advisor pushed it to 407 lines. The code is well-organized with clear section comments, so readability is not impacted today. Consider extracting the redemption advisor logic into `redemption-advisor.service.ts` in a future refactoring to stay within the 300-line guideline.

**2. Floating-point equality comparison in `buildRecommendation` (`cost-calculator.service.ts:385`)**

The check `if (cashSavings === 0)` uses strict equality on a floating-point number. `buildRecommendation` is called at line 336 with the raw, unrounded `cashSavings` value (rounding happens later at line 348). A near-zero value like `0.0000000001` would skip the "Break even" branch and fall into "Pay cash" with a negligible amount.

Practical impact is negligible — exact break-even in real usage is extremely unlikely, and the amounts displayed would round to R$0.00 anyway. If desired, a tolerance check (`Math.abs(cashSavings) < 0.01`) would be more robust.

**3. Missing UI component test (`redemption-advisor-form.tsx`)**

No co-located test file `redemption-advisor-form.test.tsx` exists. The existing `calculator-form.tsx` also lacks tests, so this follows the current project pattern. The server action and service logic are thoroughly tested, so the untested surface area is limited to React rendering and form state management. Adding component tests would improve confidence in preset handling, form validation, and result rendering.

## ✅ Positive Highlights

- **Clean layered architecture**: Schema validation (Zod) → pure calculation (service) → data access (transfer service) → server action (auth + orchestration) → UI component. Each layer has a clear single responsibility.

- **Personal cost history integration is well-implemented**: The server action correctly checks for auth, fetches the user's weighted average from transfer history filtered by program, allows manual override (skipping DB lookup), and gracefully falls back to market average for unauthenticated users.

- **Weighted averaging is the correct approach**: `getUserAverageCostPerMilheiro` uses weighted averaging by miles received, so larger transfers count proportionally more. This gives a more accurate cost basis than a simple arithmetic average.

- **Comprehensive test coverage**: 10 tests for `computeRedemptionAdvisor` (savings, pay-cash, break-even, all rating thresholds, rounding), 7 tests for `getUserAverageCostPerMilheiro` (weighted avg, no history, program filter, zero miles, rounding), 9 tests for the schema, and 8 tests for the server action (auth flow, manual override, no history, unauthenticated, errors). 100% line/function coverage on all new code.

- **Proper use of named constants**: Redemption thresholds (`REDEMPTION_THRESHOLD_EXCELLENT = 60`, `REDEMPTION_THRESHOLD_GOOD = 30`, `REDEMPTION_THRESHOLD_ACCEPTABLE = 15`) are properly extracted as named constants. `POINTS_PER_MILHEIRO` is defined locally in `transfer.service.ts` and used consistently.

- **Proper Prisma typing**: `getUserAverageCostPerMilheiro` uses `Prisma.TransferLogWhereInput` for the where clause, providing compile-time safety. The query uses `select` to fetch only needed fields.

- **Proper Pino logging**: All services use `logger.debug` with structured data (userId, program, amounts). Error paths in the server action use `logger.error` with the `err` property pattern.

- **Zod validation is thorough**: `redemptionAdvisorInputSchema` validates all fields with appropriate constraints (non-negative prices, integer miles, required program, optional user average, taxes default to 0).

- **Forward-thinking UI design**: The `CompactRedemptionResult` component is ready for embedding in flight result cards (PRD F4.10), and the `RedemptionAdvisorResultCard` is exported for reuse.

- **`readonly` properties on all interfaces**: `RedemptionAdvisorResult` and all existing interfaces use `readonly`, enforcing immutability.

- **Preset flights for quick testing**: The UI includes 4 realistic Brazilian flight presets (GRU→MIA, GRU→LIS business, GRU→EZE, VCP→SSA) that make it easy to demo and test the feature.
