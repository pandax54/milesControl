### Minor Issues

**1. Magic numbers in `rateRedemptionValue` (cost-calculator.service.ts, lines 358-364)**

The acquisition-side thresholds (12, 16, 20) are extracted into named constants (`RATING_THRESHOLD_EXCELLENT`, `RATING_THRESHOLD_GOOD`, `RATING_THRESHOLD_ACCEPTABLE`), but the redemption-side thresholds (60, 30, 15) are inline magic numbers. For consistency, these should be named constants.

Suggested fix:
```typescript
const REDEMPTION_THRESHOLD_EXCELLENT = 60;
const REDEMPTION_THRESHOLD_GOOD = 30;
const REDEMPTION_THRESHOLD_ACCEPTABLE = 15;

function rateRedemptionValue(milesValuePerK: number): PromotionRating {
  if (milesValuePerK <= 0) return 'AVOID';
  if (milesValuePerK >= REDEMPTION_THRESHOLD_EXCELLENT) return 'EXCELLENT';
  if (milesValuePerK >= REDEMPTION_THRESHOLD_GOOD) return 'GOOD';
  if (milesValuePerK >= REDEMPTION_THRESHOLD_ACCEPTABLE) return 'ACCEPTABLE';
  return 'AVOID';
}
```

**2. Magic number `1000` in `getUserAverageCostPerMilheiro` (transfer.service.ts, line 148)**

The divisor `1000` on line 148 (`totalMilesSum / 1000`) is the points-per-milheiro constant. The cost-calculator service already has `POINTS_PER_MILHEIRO = 1000`. This same magic number appears on line 7 as well (pre-existing). Consider importing the constant or defining one locally.

Suggested fix:
```typescript
const POINTS_PER_MILHEIRO = 1000;
// ...
const avgCostPerMilheiro = totalCostSum / (totalMilesSum / POINTS_PER_MILHEIRO);
```

**3. Floating-point equality comparison in `buildRecommendation` (cost-calculator.service.ts, line 380)**

The check `if (cashSavings === 0)` uses strict equality on a floating-point number that results from subtraction. In practice, this is mitigated because the caller rounds to 2 decimal places before calling `buildRecommendation` -- wait, actually `buildRecommendation` is called BEFORE rounding (line 331), so `cashSavings` at this point is the raw unrounded value. The rounding happens in the return statement (line 341). This means a near-zero value like `0.0000000001` would fall through to the "Pay cash" branch.

However, the practical impact is negligible since the break-even test (line 526-536) works because the inputs are engineered to produce exactly 0. In real usage, exact break-even is extremely unlikely. This is a theoretical edge case only.

**4. `where` typed as `Record<string, unknown>` (transfer.service.ts, line 114)**

The Prisma `where` clause is typed as `Record<string, unknown>` instead of using Prisma's generated `TransferLogWhereInput` type. This bypasses Prisma's type safety. While it works at runtime, using the generated type would catch errors at compile time.

Suggested fix:
```typescript
import type { Prisma } from '@/generated/prisma/client';
// ...
const where: Prisma.TransferLogWhereInput = {
  userId,
  totalCost: { not: null },
  costPerMilheiro: { not: null },
  milesReceived: { gt: 0 },
};
if (program) {
  where.destProgramName = program;
}
```

**5. Missing UI component test (redemption-advisor-form.tsx)**

There is no test file `redemption-advisor-form.test.tsx` co-located with the component. The existing `calculator-form.tsx` also lacks tests, so this follows the current pattern. However, the project standard specifies co-located test files. The server action and service logic are well-tested, so the untested surface area is limited to React rendering and state management.

**6. `cost-calculator.service.ts` file is 401 lines**

The file exceeds the 300-line guideline for a class/module. It was already close to the limit after task 3.6/3.7, and the ~100 lines added for the redemption advisor pushed it over. Consider extracting the redemption advisor logic into its own file (`redemption-advisor.service.ts`) in a future refactoring.
